1Databricks Platform
Easy
What is the Lakehouse architecture, and what does Databricks add on top of plain Spark?
Model answer

A Lakehouse merges the low-cost, open storage of a data lake with the ACID transactions, schema enforcement, and BI performance of a warehouse — Delta Lake is the storage layer that makes this possible. Databricks adds the managed control plane: cluster orchestration, Unity Catalog for governance, Workflows for scheduling, and Photon for vectorized execution.

Pivot to your storyYou can immediately follow this with your Bronze/Silver/Gold/Platinum extension — proof you don't just know the definition, you've operationalized it.

Medium
Walk me through your Bronze/Silver/Gold/Platinum design. Why a fourth layer?
Model answer

Bronze = raw landed data, Silver = cleaned/conformed/deduped (your SCD2 + DQ checks live here), Gold = business-aggregated/reporting-ready. Platinum was your addition for cross-domain, audit-grade outputs — e.g. LCA/TCFD reporting tables that need IPE documentation and have stricter lineage/PII requirements than a normal Gold table.

Why it lands wellIt shows you can extend a textbook pattern when the compliance reality (ESG audit trails) demands it, rather than cargo-culting Medallion as-is.

Hard
All-purpose clusters vs Job clusters vs SQL Warehouses — how do you choose, and what would you run your 25-feed daily ingestion on?
Model answer

All-purpose clusters are for interactive dev/notebooks — expensive to leave running, billed per DBU even idle. Job clusters spin up per job and terminate after, much cheaper for scheduled production pipelines. SQL Warehouses (serverless or classic) are tuned for BI/SQL workloads with auto-scaling and result caching.

For 25 daily ingestion feeds: job clusters, one per workflow (or a shared job cluster across tasks in a single workflow run to amortize startup time), autoscaling 2–8 nodes, with a cluster policy to cap node type/spend.

Brutal
Your Bronze-to-Silver job runs 4× slower at only 2× the data volume. Walk me through your diagnosis, live, start to finish.
Model answer — narrate it like a runbook
  • 1. Spark UI → Stages tab: find the stage with the longest duration and check task-level skew (max task time vs median).
  • 2. Check shuffle read/write size — did a join/groupBy explode partition count or did a broadcast threshold get exceeded silently (table grew past spark.sql.autoBroadcastJoinThreshold)?
  • 3. Check partition count vs cluster cores — too few partitions = under-utilization; too many = scheduling overhead.
  • 4. Check for small-file problem on the source side (common in your CSV/Parquet ingestion) — many small files inflate listing + task overhead non-linearly with row count.
  • 5. Check AQE is enabled — without it, a stale partition count from before the 2× growth could be causing it.
  • 6. Check GC time / spill to disk in executor metrics — memory pressure causes disproportionate slowdown, not linear.

Land the answer on: "non-linear slowdown is almost always skew, spill, or a broadcast threshold being crossed — not just 'more data'."

2PySpark Internals
Easy
Explain lazy evaluation and the difference between a transformation and an action.

Transformations (select, filter, withColumn, join) build up a logical plan (DAG) without executing. Nothing runs until an action (show, count, write, collect) triggers the Catalyst optimizer to build a physical plan and execute. This lets Spark optimize the whole chain (predicate pushdown, projection pruning) before touching data.

Medium
Narrow vs wide transformations — and how does this relate to your schema-drift framework across 10+ source feeds?

Narrow transformations (filter, map, withColumn) operate per-partition with no data movement. Wide transformations (groupBy, join, distinct, repartition) require a shuffle — data moves across the network/disk, and this is where most performance problems live.

PivotTie this to schema drift: comparing incoming vs target schema is a narrow, cheap metadata operation; the expensive part is the downstream wide merge once you reconcile columns — call this out to show you think about cost, not just correctness.

Hard
Your BOM hierarchy explosion uses iterative joins across 10M+ material keys. How do you stop it blowing up the lineage/DAG, and how do you detect cycles?

Each loop iteration adds to Spark's logical plan lineage; after enough iterations the planner itself becomes the bottleneck (and you risk StackOverflow on plan analysis). Fix: df.checkpoint() (with a reliable checkpoint dir) or df.localCheckpoint() every N iterations to truncate lineage and materialize the DataFrame. For cycle detection, track visited node IDs in a running "seen" set/DataFrame and break (or flag) when a child reappears as its own ancestor — otherwise the explosion never terminates on bad master data.

Brutal
Your UDF-based PII masking layer works but is slow at scale. Why, and how would you redesign it?

Python UDFs serialize each row out of the JVM into a Python process (and back), breaking Catalyst's whole-stage codegen and Photon vectorization — that round trip dominates cost at scale. Fix path: replace with native Spark SQL functions (sha2/hash for irreversible masking, regexp_replace, substring tricks for partial masking, or aes_encrypt/format-preserving approaches) wherever logic permits; if truly custom logic is needed, move to a Pandas UDF (vectorized, batches via Arrow) as a middle ground; for governance-grade masking, also evaluate Unity Catalog's native column masks / dynamic views so the rule lives in the catalog, not scattered across notebooks.

3Spark / Delta Optimization
Easy
What is partition pruning and predicate pushdown?

Predicate pushdown moves filter conditions as close to the data source as possible so unneeded rows are never read into Spark. Partition pruning is the specific case where the filter is on a partition column, so entire partition directories are skipped at the file-listing stage — no scan at all.

Medium
Explain Z-Ordering and when you'd run OPTIMIZE on your Delta tables.

OPTIMIZE compacts many small files into fewer larger ones (solving the small-file problem from your daily 25-feed ingestion). Z-Ordering co-locates related data by clustering rows on the Z-order columns (usually high-cardinality filter columns like material_key or source_id), so data-skipping statistics become much more effective for those predicates. Run it on hot Silver/Gold tables on a schedule, not on Bronze where you're still appending raw landings.

Hard
What does Adaptive Query Execution (AQE) actually change at runtime?

AQE re-optimizes the physical plan mid-query using actual shuffle statistics instead of static estimates: it coalesces post-shuffle partitions that turned out tiny, converts a sort-merge join to a broadcast join if a table turns out smaller than expected, and splits/handles skewed partitions automatically (skew join optimization) by splitting the oversized partition and replicating the matching side.

Brutal
One join key holds 90% of rows (e.g. a default/null material_key). AQE skew-join handling isn't enough. Design the full fix.
  • Isolate the hot key: split the DataFrame into skewed-key rows and normal rows.
  • Salt the skewed side: add a random suffix (e.g. 0–N) to the join key on the large side; explode the smaller side N times with each salt value so every salted row has a match.
  • Union the results of the salted join and the normal join back together.
  • Validate: re-check Spark UI task time distribution — max task time should drop close to median.
  • Root-cause it too — if 90% share one key because of bad upstream defaulting (e.g. unmapped material_key), fix the data quality issue so you're not permanently engineering around it.
4Workflow / Orchestration
Easy
What's a Databricks Workflow, and what's a "task" within it?

A Workflow is Databricks' native job scheduler — a DAG of tasks where each task can be a notebook, JAR, Python script, SQL file, or dbt model, with dependencies, retries, and parameters defined between them. It replaces needing an external orchestrator for purely Databricks-internal pipelines.

Medium
When do you orchestrate from Azure Data Factory vs native Databricks Workflows?

ADF when the pipeline spans non-Databricks systems — copying from on-prem SQL Server, SFTP, REST APIs, triggering Synapse, or when business users need a low-code view. Databricks Workflows when the entire pipeline lives inside the Lakehouse — better task-level observability, lower latency between tasks, and native integration with cluster reuse/job clusters for cost control.

Hard
You built an ingestion monitoring framework that alerts owners by email on failure/delay across 25 files and 5 apps. How would you formalize that into the orchestration layer rather than custom code?

Replace ad-hoc email logic with: task-level retry policies + timeout config in Workflows; a dedicated "alerting" task wired via on_failure webhook (or system table queries on job run status) feeding a notification channel (email/Teams/PagerDuty); SLA definitions per task using expected duration thresholds; and a dashboard over the system.lakeflow job-run tables for owners to self-serve instead of relying on push alerts alone.

Brutal
Design end-to-end orchestration for 25 daily ESG feeds across 5 apps with backfill capability and idempotency guarantees.
  • Idempotency: every Bronze write keyed by (source, business_date, batch_id) using MERGE or overwrite-by-partition, never blind append, so reruns don't duplicate.
  • Backfill: parameterize the workflow by a date range; a control table tracks watermark/last-successful-load per feed so backfills can target just the gap.
  • Dependency management: per-app DAGs fan into a shared Silver consolidation task gated on all 5 apps' Bronze tasks succeeding (or partial-success policy with explicit allow-list).
  • SLA + alerting as above, plus a dead-letter pattern for rows that fail DQ checks instead of blocking the whole batch.
  • Auditability: every run logs to a control/metadata table (rows in, rows rejected, duration) — this is your IPE documentation made automatic instead of manual.
5Azure Cloud (your bridge topic)
Easy
What's the Azure equivalent of an S3 bucket, and how is it structured?

ADLS Gen2 (Azure Data Lake Storage Gen2) — built on Blob Storage with a hierarchical namespace. A Storage Account is the top-level resource (akin to an AWS account-scoped bucket namespace); inside it, Containers hold your actual folder/file hierarchy, comparable to an S3 bucket.

Medium
How does Unity Catalog authenticate to ADLS instead of S3?

Instead of an IAM Role + instance profile, UC uses a Storage Credential backed by an Azure Managed Identity (Access Connector for Databricks) with RBAC role assignment (Storage Blob Data Contributor, typically) on the target container, then an External Location mapping that credential to an ADLS path. Conceptually identical to your S3 + IAM Role setup — same 3-piece pattern (credential → location → grant).

Hard
Explain VNet injection / Private Endpoints for a Databricks workspace, and why a regulated ESG client would require it.

VNet injection deploys the Databricks compute plane into the customer's own virtual network instead of a Databricks-managed one, giving control over subnets, NSGs (network security groups), and route tables. Private Endpoints let the workspace reach ADLS/Key Vault over Microsoft's backbone instead of the public internet. For ESG/sustainability data with audit requirements (your IPE documentation work), this closes the "data never traverses the public internet" compliance gap that auditors specifically check for.

Brutal
Design a secure landing zone in Azure for the exact regulated ESG workload you described from your CV — name every component.
  • Storage: ADLS Gen2 with hierarchical namespace, lifecycle policies for Bronze cold-tiering.
  • Identity: Service Principal / Managed Identity per environment, no shared credentials, scoped via Azure RBAC.
  • Secrets: Azure Key Vault, surfaced into Databricks via Key Vault-backed secret scopes (no PATs in code).
  • Network: VNet-injected workspace, Private Endpoints to storage/Key Vault, NSGs denying public egress by default.
  • Governance: Unity Catalog with catalogs per environment, External Locations scoped to least-privilege paths, column masks for the 15+ PII columns you previously handled via UDFs.
  • Audit: Unity Catalog audit logs + system tables feeding a compliance dashboard — your manual IPE docs become queryable lineage.
6CI/CD & GitHub
Easy
Why version notebooks/code in Git instead of relying on Databricks' built-in revision history?

Built-in revision history is per-notebook, has no branching/PR review, and isn't portable across workspaces (dev/test/prod). Git gives you code review, branch protection, a single source of truth deployable to multiple workspaces, and CI hooks (tests, linting) that catch issues before they hit a shared cluster.

Hard
What are Databricks Asset Bundles (DABs) and why are they better than ad-hoc deployment scripts?

DABs are a YAML-based IaC layer (databricks.yml) describing jobs, clusters, and notebooks as code, with per-target overrides (dev/staging/prod) for cluster size, catalog, and parameters. databricks bundle validate/deploy/run gives you a repeatable, diffable deployment instead of bespoke shell scripts — same philosophy as Terraform but scoped to the Databricks workspace.

Brutal
Design full CI/CD for your schema-drift framework: PR → tests → dev → staging → prod, with rollback.
  • PR stage: unit tests on the schema-comparison logic (using chispa/pytest with a local Spark session) + linting; block merge on failure.
  • Dev deploy: DAB deploy to a dev target/catalog on merge to a develop branch; run against synthetic/sampled data.
  • Staging: deploy to a staging catalog, run against a masked copy of real data volumes, validate DQ check pass rates and runtime regressions against a baseline.
  • Prod promotion: manual approval gate (given this is PII/ESG-compliance-sensitive data) before bundle deploy --target prod.
  • Rollback: tag every prod deploy to a Git commit SHA; rollback = redeploy the previous tagged bundle, plus Delta's time travel (RESTORE TABLE ... TO VERSION AS OF) if bad data already landed.
7Service Principals
Easy
What is a service principal and why shouldn't production jobs run under a personal access token?

A service principal is a non-human identity for an application/automation to authenticate as — it has its own permissions, isn't tied to a person, and doesn't break when that person leaves or rotates their password. PATs tied to a person are a single point of failure, hard to audit ("was this job or Bob?"), and a compliance red flag — exactly the kind of thing your PII governance work would flag.

Medium
How do you grant a service principal access to a Unity Catalog catalog/schema, and what's the least-privilege pattern?

Add the SP as a UC principal, then GRANT USE CATALOG, USE SCHEMA, SELECT ON TABLE ... scoped to exactly the catalog/schema it needs — never catalog-wide admin. Best practice: grant to a group the SP belongs to, not the SP directly, so access reviews scale and you can swap SPs without re-auditing every grant.

Hard
ADF needs to trigger a Databricks job via REST API. How do you wire the authentication securely?

Register an Azure AD app (the service principal), grant it the minimum Databricks workspace permission (Job Run permission on that specific job, not admin), store its client secret in Azure Key Vault, and have ADF's Databricks linked service authenticate via Azure AD token (not a static PAT) — ADF natively supports SP-based auth for the Databricks Linked Service. Rotate the secret on a schedule via Key Vault, never hardcode it in the pipeline JSON.

Brutal
Design the full SP lifecycle/rotation model across dev/test/prod for a regulated, audited ESG platform.
  • One SP per environment per workload class (ingestion SP, transformation SP, BI-read SP) — never one SP for everything.
  • Secrets in environment-scoped Key Vaults, referenced via Key Vault-backed Databricks secret scopes — app code never sees the raw secret.
  • Automated rotation (Key Vault rotation policy + an Azure Function or pipeline step that updates the secret and notifies dependent jobs) on a fixed cadence (e.g. 90 days).
  • Access reviews: quarterly audit of SP-to-UC grants, tied to the same audit trail you built for PII columns (IPE-style evidence).
  • Break-glass plan: what happens if a secret leaks — revoke in Key Vault, the SP's grants are scoped tight enough that blast radius is one schema, not the catalog.
8Unity Catalog
Easy
Explain Unity Catalog's three-level namespace.

catalog.schema.table — catalog is the top-level container (often per environment or business unit), schema groups related tables (a domain, like "esg_reporting"), and table is the object itself. This replaces Hive Metastore's two-level database.table and adds a governance boundary at the catalog level that HMS never had.

Medium
You migrated 15+ Hive Metastore framework codes to Unity Catalog. What actually breaks in that migration?

Three-part naming forces every hardcoded db.table reference to become catalog.db.table; HMS-style implicit cluster-level permissions don't exist in UC, so every job's SP/user needs explicit grants; external table paths need a registered External Location + Storage Credential instead of relying on the cluster's instance profile; and any logic depending on HMS-specific metastore functions needs auditing since UC enforces stricter SQL compliance.

Hard
You built PII masking via custom PySpark UDFs at the table level. How would UC's native row/column-level security change that design?

UC supports column masks (a function applied automatically based on the querying principal's group membership) and row filters, defined once at the catalog/schema level instead of duplicated logic inside every notebook. A finance analyst and a data engineer can query the same physical table and see different values for the same column — no separate masked copy needed, no risk of someone bypassing the masking notebook and querying Bronze raw.

Brutal
Design full UC governance for a multi-team ESG platform at 10× your current scale.
  • Catalogs per environment (dev/staging/prod) — never share a prod catalog with dev workloads.
  • Schemas per business domain (csr, lca, tcfd, sustainability) so access can be domain-scoped, not table-by-table.
  • Access via Azure AD groups synced into UC, never individual user grants — onboarding/offboarding becomes a group membership change, not a grant audit.
  • Column masks for all PII at the schema level (replacing your per-table UDF approach), enforced regardless of which tool queries the table.
  • Lineage (UC's built-in lineage graph) replacing manual IPE documentation — auditors can trace a Gold metric back to its Bronze source automatically.
  • What you'd do differently: introduce UC from day one of a project rather than migrating HMS code later — most of the 15+ framework-code rewrite pain was technical debt from not starting on UC.

Write the SCD Type 2 MERGE for the Employee Dimension you described on your CV. 
MERGE INTO dim_employee AS target
USING staged_employee_updates AS source

Hard
Recursive CTE to flatten a multi-level BOM (parent_id, child_id, qty) — your PySpark hierarchy job, in SQL. 
WITH RECURSIVE bom_explosion AS (
  -- anchor: top-level assemblies (no parent)
  SELECT material_id AS root_id, material_id, parent_id, qty,
         1 AS level, qty AS total_qty,
         ARRAY(material_id) AS path
  FROM bom WHERE parent_id IS NULL

  UNION ALL

  -- recursive step: walk down to children
  SELECT e.root_id, b.material_id, b.parent_id, b.qty,
         e.level + 1, e.total_qty * b.qty,
         e.path || ARRAY(b.material_id)
  FROM bom b
  JOIN bom_explosion e ON b.parent_id = e.material_id
  WHERE NOT ARRAY_CONTAINS(e.path, b.material_id)  -- cycle guard
)
SELECT * FROM bom_explosion ORDER BY root_id, level;

Talking pointMention that Spark SQL doesn't support recursive CTEs natively (this is ANSI SQL / SQL Server / Postgres syntax) — that's exactly why you built the iterative-join version in PySpark instead. Knowing the boundary between engines is the point of this question.

Brutal
Find the longest streak of consecutive "Pass" DQ checks per source feed, and flag feeds with ≥3 consecutive "Fail". 
-- classic gaps-and-islands: group consecutive same-status rows
WITH flagged AS (
  SELECT source_feed, check_date, status,
    ROW_NUMBER() OVER (PARTITION BY source_feed ORDER BY check_date)
    - ROW_NUMBER() OVER (PARTITION BY source_feed, status ORDER BY check_date)
    AS grp
  FROM dq_checks
),
islands AS (
  SELECT source_feed, status, grp, COUNT(*) AS streak_len,
         MIN(check_date) AS streak_start
  FROM flagged
  GROUP BY source_feed, status, grp
)
SELECT source_feed,
       MAX(CASE WHEN status = 'Pass' THEN streak_len END) AS longest_pass_streak,
       MAX(CASE WHEN status = 'Fail' AND streak_len >= 3 THEN streak_len END) AS sla_breach_streak
FROM islands
GROUP BY source_feed;

Talking pointThis is the "row_number minus row_number" trick — explain it as you write it: the difference between the overall sequence position and the within-status sequence position stays constant exactly for the length of a consecutive run. Ties directly to your 10+ integrity-check ingestion validation work.

Brutal
Implement the recursive BOM explosion iteratively, with checkpointing and a cycle guard, at 10M+ row scale. 
spark.sparkContext.setCheckpointDir("/mnt/checkpoints/bom")

frontier = (bom_df
    .filter(F.col("parent_id").isNull())
    .select(F.col("material_id").alias("root_id"),
            "material_id", F.lit(1).alias("level"),
            F.array("material_id").alias("path")))

result = frontier
MAX_DEPTH = 20

for depth in range(MAX_DEPTH):
    next_level = (frontier.alias("f")
        .join(bom_df.alias("b"), F.col("b.parent_id") == F.col("f.material_id"))
        # cycle guard: skip if child already in this path
        .filter(~F.array_contains(F.col("f.path"), F.col("b.material_id")))
        .select("f.root_id", F.col("b.material_id"),
                (F.col("f.level") + 1).alias("level"),
                F.array_union("f.path", F.array("b.material_id")).alias("path")))

    if next_level.rdd.isEmpty():    # terminate when frontier is exhausted
        break

    next_level = next_level.checkpoint()  # truncate lineage every loop
    result = result.unionByName(next_level)
    frontier = next_level

Talking pointThree things to call out unprompted: (1) checkpointing every iteration to stop lineage blow-up, (2) the array-contains cycle guard so bad master data can't infinite-loop, (3) a hard MAX_DEPTH safety valve regardless — never trust the data alone to terminate the loop.

def flatten(d, parent_key="", sep="."):
    items = {}
    for k, v in d.items():
        new_key = f"{parent_key}{sep}{k}" if parent_key else k
        if isinstance(v, dict):
            items.update(flatten(v, new_key, sep))
        else:
            items[new_key] = v
    return items

import time, random, functools

def retry(max_attempts=4, base_delay=1.0):
    def decorator(fn):
        @functools.wraps(fn)
        def wrapper(*args, **kwargs):
            for attempt in range(1, max_attempts + 1):
                try:
                    return fn(*args, **kwargs)
                except Exception as e:
                    if attempt == max_attempts:
                        raise
                    delay = base_delay * (2 ** (attempt - 1)) + random.uniform(0, 0.5)
                    print(f"Attempt {attempt} failed: {e}. Retrying in {delay:.1f}s")
                    time.sleep(delay)
        return wrapper
    return decorator

Hard
Recursively diff two nested JSON schemas and return added / removed / type-changed fields. 
def diff_schema(old, new, path=""):
    changes = {"added": [], "removed": [], "type_changed": []}
    old_keys, new_keys = set(old), set(new)

    for key in new_keys - old_keys:
        changes["added"].append(f"{path}.{key}")
    for key in old_keys - new_keys:
        changes["removed"].append(f"{path}.{key}")

    for key in old_keys & new_keys:
        old_val, new_val = old[key], new[key]
        if isinstance(old_val, dict) and isinstance(new_val, dict):
            nested = diff_schema(old_val, new_val, f"{path}.{key}")
            for k in changes: changes[k] += nested[k]
        elif type(old_val) != type(new_val):
            changes["type_changed"].append(f"{path}.{key}: {type(old_val).__name__} -> {type(new_val).__name__}")

    return changes
Brutal
Build a simplified version of your DDL auto-generation script: infer a schema across multiple CSVs for the same logical table and resolve type conflicts. 
import pandas as pd
from pathlib import Path

TYPE_PRECEDENCE = ["int64", "float64", "bool", "datetime64[ns]", "object"]
SPARK_TYPE_MAP = {"int64": "BIGINT", "float64": "DOUBLE",
                   "bool": "BOOLEAN", "datetime64[ns]": "TIMESTAMP",
                   "object": "STRING"}

def widen(t1, t2):
    # pick whichever type sits later in the precedence list (safer / wider)
    return t1 if TYPE_PRECEDENCE.index(t1) >= TYPE_PRECEDENCE.index(t2) else t2

def infer_merged_schema(csv_dir: str) -> dict:
    merged = {}
    for file in Path(csv_dir).glob("*.csv"):
        df = pd.read_csv(file)
        for col, dtype in df.dtypes.items():
            dtype = str(dtype)
            if col not in merged:
                merged[col] = dtype
            elif merged[col] != dtype:
                merged[col] = widen(merged[col], dtype)
    return merged

def generate_ddl(table_name: str, schema: dict) -> str:
    cols = ",\n  ".join(f"{c} {SPARK_TYPE_MAP[t]}" for c, t in schema.items())
    return f"CREATE TABLE {table_name} (\n  {cols}\n) USING DELTA;"
