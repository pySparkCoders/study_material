Liquid Clustering

How automatic data organisation replaces manual partitioning strategies — and when to actually use it.

What problem does it solve?
Traditional Hive-style partitioning (e.g. PARTITION BY date) was designed for HDFS in 2009. It works by splitting data into separate folders on disk. This is fast when you always query by that same column — but collapses in three real-world situations: (1) analysts start filtering on a different column than the one you partitioned by, (2) your partition column has too many distinct values (a "partition explosion"), or (3) your access patterns change over time. Every time this happens with traditional partitioning, you either suffer full-table scans or do a full rewrite of your data.

Liquid Clustering (GA in Databricks Runtime 13.3+) replaces this with an automatic, incremental layout optimiser that works without requiring you to pre-decide your query patterns.
💡
The core idea: Instead of physically splitting data into folders, Liquid Clustering co-locates related rows within the same Parquet files using a Hilbert space-filling curve — a mathematical path through multi-dimensional space that preserves locality. Rows with similar clustering key values land in the same file, making file-level data skipping highly effective.
How it works technically
1

You define clustering keys

CREATE TABLE events CLUSTER BY (user_id, event_date) — you specify which columns your queries filter on most. These become the clustering dimensions.

2

New data writes with partial clustering

Incoming data is partially clustered at write time. Fully optimised clustering happens lazily when you run OPTIMIZE table_name.

3

Hilbert curve + tree balancer organises files

The Hilbert curve maps multi-dimensional key values into a 1D ordering. A tree-based layout balancer then groups rows into balanced Parquet files. This handles high-cardinality keys and skewed distributions automatically — something Z-ORDER cannot do.

4

Delta's data skipping uses min/max stats

Because related rows are now co-located, when your query filters on user_id = 12345, Delta checks per-file min/max stats and skips every file that can't contain that value. With good clustering, 95%+ of files can be skipped.

Liquid Clustering vs Partitioning vs Z-ORDER
Partitioning
  • Stable, low-cardinality columns (<1,000 values)
  • All queries filter on the same column
  • Best elimination: entire directory skipped
  • Changing the partition key = full rewrite
  • Can combine with Z-ORDER for secondary filters
Liquid Clustering
  • High-cardinality columns (millions of values)
  • Access patterns change over time
  • Best elimination: file-level via stats
  • Changing keys: ALTER TABLE — only future writes
  • Cannot combine with Z-ORDER or partitions
⚠
Mutual exclusivity: You CANNOT use CLUSTER BY on a table that already has partition columns, and you CANNOT run OPTIMIZE ZORDER BY on a CLUSTER BY table. Both Z-ORDER and Liquid Clustering use Hilbert curves — the engine can't reconcile two layout strategies simultaneously.
📋
The decision rule: If your filter column has <1,000 distinct values and never changes → stick with partitioning. If cardinality is high or access patterns evolve → use Liquid Clustering. For new tables, default to Liquid Clustering.
Migration reality check (the 2TB question)
Migrating a 2 TB partitioned table to Liquid Clustering requires a full data rewrite. This means: compute cost for the rewrite job, a period of downtime or dual-write complexity, and updates to any downstream pipelines that rely on the physical file path structure (e.g. external Hive readers expecting /date=2024-01-01/ folders).

Before migrating, calculate the payback period: cost of full-partition scans on non-partition columns × weekly query count vs one-time migration compute cost. If payback is < 4 weeks, migrate. If the table is stable and access patterns haven't changed, leave it alone.

Knowledge check

You have a 2 TB table partitioned by region (50 values). 40% of queries now filter by user_id (5M distinct values). What do you do? ⌄
Migrate to Liquid Clustering — but quantify first. Calculate the compute cost of full-partition scans (40% of queries scanning all 50 partitions × 2 TB) vs the one-time migration rewrite cost. If analysts keep adding dashboards on non-partition columns, CLUSTER BY (user_id, region) will eliminate both scan penalties going forward. Mention the payback period.
Why can't you run OPTIMIZE ZORDER BY on a CLUSTER BY table? ⌄
Both use Hilbert space-filling curves to co-locate data. Z-ORDER applies the curve manually per OPTIMIZE run with static results. Liquid Clustering adds a tree-based balancer on top and applies incrementally. A Delta table can only carry one layout strategy in its metadata — having both would create conflicting reorganisation semantics on every OPTIMIZE run.
You need to change the clustering key from user_id to account_id. What happens? ⌄
With Liquid Clustering: run ALTER TABLE t CLUSTER BY (account_id). Only future writes use the new key — existing data is gradually re-clustered by subsequent OPTIMIZE runs. No full rewrite required. This is the key operational advantage over partitioning where changing the partition key requires a complete data migration.
Lesson 02 / 10SparkPerformance

How One SQL Line Becomes 4,000 Parallel Tasks

The complete Catalyst → Tungsten → AQE pipeline — and why understanding it is the difference between listing optimisation tips and actually fixing slow jobs.

The full compilation pipeline
1

Parsing → Unresolved Logical Plan

The SQL text is parsed into an Abstract Syntax Tree (AST). Column names and table references exist but are not yet validated against the Catalog. The result is an "unresolved" logical plan.

2

Analysis → Resolved Logical Plan

The Analyzer resolves column names, table references, and function calls against the Spark Catalog (Hive Metastore or Unity Catalog). This produces a resolved logical plan with concrete types and references.

3

Catalyst Optimizer → Optimised Logical Plan

A rule-based optimizer applies dozens of rules in batches: constant folding (evaluate 1 + 1 at plan time), predicate pushdown (move filters as early as possible), column pruning (remove columns not referenced), and join reordering (put smaller tables first). This is Catalyst's logical optimisation phase.

4

Physical Planning → Physical Plan

Catalyst generates multiple candidate physical plans (e.g. BroadcastHashJoin vs SortMergeJoin for each join) and uses a cost model to select the best one. This is where join strategy decisions are made at plan time.

5

Tungsten Code Generation → JVM Bytecode

Whole-stage code generation compiles the physical plan into a single tight JVM bytecode loop, eliminating virtual dispatch overhead. Operations that would normally call multiple virtual methods are fused into a single tight loop — this is where Spark gets its raw speed on modern CPUs.

6

AQE — Runtime Adaptation

Adaptive Query Execution re-optimises the plan at runtime using actual statistics gathered after each stage: partition coalescing (merging small shuffle partitions), skew join handling (splitting large partitions), and join strategy switching (downgrading SortMergeJoin to BroadcastHashJoin if a partition turns out small enough).

Key AQE configurations
SettingDefaultWhat it controls
spark.sql.adaptive.enabledtrue (DBR)Master AQE toggle
spark.sql.adaptive.skewJoin.enabledtrueAutomatic skew join splitting
spark.sql.adaptive.skewJoin.skewedPartitionFactor5×Threshold to detect skewed partitions vs median

What interviewers actually evaluate

Mid
Correctly explains what each technique does, but can't say which one to apply first without seeing the job. Analysis paralysis.
Senior

Principal
Instruments pipelines proactively with stage-level metrics before they become slow. Owns the SLA rather than reacting to it.
💡
The Spark UI diagnostic order: Stages tab (find the longest stage) → Task duration distribution (check for skew) → Shuffle read/write size (check for missing pushdown) → SQL tab physical plan (verify join strategy and predicate pushdown) → Executor GC time (check for memory pressure).

Knowledge check

A job has 200 shuffle partitions, each 10 GB. What's the problem and fix? ⌄
Problem: 10 GB partitions are far too large — recommended target is ~128 MB per partition. This means tasks are doing too much work per task, are hard to parallelize, and may spill to disk.

Fix: Increase spark.sql.shuffle.partitions to 2 TB / 128 MB ≈ 16,000. BUT first check if the bottleneck is actually skew (a few partitions huge, most small) vs uniform large partitions — these require different fixes.
What does AQE's skew join handling actually do at runtime? ⌄
After the first shuffle stage completes, AQE reads the actual partition size statistics. If any partition is more than skewedPartitionFactor (default 5×) the median size AND larger than skewedPartitionThresholdInBytes (default 256 MB), AQE automatically splits that partition into smaller sub-partitions and replicates the corresponding rows from the other table to join against each sub-partition. The join completes in more, smaller tasks instead of one massive task.

Partition Pruning vs Data Skipping

Two optimisation layers — directory level and file level — and why understanding the difference wins interviews.

Two different layers of elimination
Partition Pruning (directory level)
  • Operates on the folder structure of partitioned tables
  • When filter matches partition column, entire directories are skipped without opening any files
  • E.g. WHERE date = '2024-01-01' on a date-partitioned table skips every other date folder
  • Zero file I/O for eliminated partitions
  • Coarse-grained — only works when filter is on the exact partition column
Data Skipping (file level)
  • Operates on per-file statistics stored in the Delta transaction log
  • Delta records min/max value for each column in each Parquet file
  • When a filter can't possibly match a file's min/max range, the file is skipped
  • Works on any column — not just partition columns
  • Fine-grained — depends on how well data is physically co-located in files
💡
They work together: Partition pruning eliminates directories first. Data skipping then eliminates Parquet files within the surviving directories. Liquid Clustering and Z-ORDER improve data skipping by ensuring rows with similar key values land in the same files — maximising the chance that a file's min/max range is narrow and can be fully skipped.
Statistics written to the delta log
StatStored per file for each columnEnables
minValuesSmallest value in the columnSkip if filter value < min
maxValuesLargest value in the columnSkip if filter value > max
nullCountNumber of null valuesSkip if filter is NOT NULL and nullCount = rowCount
numRecordsTotal rows in the fileUsed for cardinality estimation by planner
📋
Default stats depth: Delta collects stats for the first 32 columns (by position). Columns beyond position 32 have no stats and can't benefit from data skipping — a common gotcha on wide tables. Reorder important filter columns to the left.

Knowledge check

A Delta table has 500 files. A query filters WHERE amount > 1000. Without clustering, how many files are read? After clustering by amount? ⌄
Without clustering: amount values are scattered randomly across all files. The min/max per file likely spans the full range. Spark reads all 500 files.

After clustering by amount: Files containing low amounts (all < 1000) have a maxValue < 1000 and are skipped entirely. In a well-clustered table, 70-95% of files may be skipped — Spark reads only the files whose max amount exceeds 1000.
Your table has 200 columns. Column 185 is heavily filtered in queries. Data skipping isn't working. Why? ⌄
Delta only collects min/max statistics for the first 32 columns by default (configurable via delta.dataSkippingNumIndexedCols). Column 185 has no stats, so data skipping can't eliminate any files based on it. Fix: either increase delta.dataSkippingNumIndexedCols to cover the important column, or reorder the table schema to put frequently filtered columns in the first 32 positions.
Lesson 04 / 10Delta Lake

Schema Evolution in Delta Lake

The follow-up questions that separate mid-level from senior answers — and why "just turn on mergeSchema" is never the complete answer.

Delta's default behaviour
By default, Delta Lake enforces strict schema enforcement. If your incoming DataFrame has a column that doesn't exist in the table, or a column with an incompatible type, the write fails immediately with a AnalysisException. This is intentional — it prevents accidental schema corruption from propagating silently into your lakehouse.
Schema change types and how Delta handles each
Change typeAutomatic supportHow to handle
Add a new column✅ With mergeSchemadf.write.option("mergeSchema","true") or table property
Rename a column❌ Not automaticALTER TABLE t RENAME COLUMN old TO new
Change column type (widening)⚠ Partiallyint→long is safe; string→int requires explicit CAST
Drop a column❌ Not automaticALTER TABLE t DROP COLUMN col
Reorder columns✅ Delta handles by nameParquet is columnar — column order doesn't affect reads
⚠
The senior interview trap: Saying "just use mergeSchema=true" without discussing backward/forward compatibility. If you add a column, downstream consumers using SELECT * now get an extra column they may not handle. If you drop a column, pipelines referencing it break. The senior answer includes: assess impact on downstream consumers before any schema change.
The remediation framework
1

Identify the drift

Read the error message or compare incoming schema vs table schema programmatically: spark.read.format("delta").load(path).schema vs incoming_df.schema.

2

Classify the change

Is it additive (new column), structural (rename/type change), or destructive (drop)? Each requires a different response.

3

Assess downstream impact

Who reads this table? Do they use SELECT * (risky) or named columns (safer)? Check Unity Catalog lineage or query the system.access.audit log to find consumers.

4

Apply the appropriate fix

Additive → enable mergeSchema for this write. Structural → explicit DDL + coordinate with consumers. Destructive → deprecation period + consumer migration first.

5

Update data contracts & documentation

Update the schema documentation in Unity Catalog, notify downstream owners, and update any data contracts (expected column names, types, nullability).

Knowledge check

A source renames a column from customer_name to full_name. Your pipeline breaks. Walk through your exact fix. ⌄
This is a structural change — mergeSchema won't help (it only adds columns). Steps:

1. In the write path, add a transformation: .withColumnRenamed("full_name", "customer_name") to maintain backward compatibility until downstream is updated.
2. In Delta, run ALTER TABLE t RENAME COLUMN customer_name TO full_name to align the table schema.
3. Update downstream consumers to reference full_name.
4. Remove the withColumnRenamed transformation once all consumers are updated.

Never rename the column in Delta before updating consumers — you'll break them.
What's the difference between backward compatibility and forward compatibility in schema evolution? ⌄
Backward compatibility: New schema can read data written by old schema. Example: adding a nullable column is backward compatible — old data just has nulls for the new column.

Forward compatibility: Old schema can read data written by new schema. Example: if old readers ignore unknown columns, adding a column is forward compatible. If they enforce strict schema, it's not.

The senior answer: always check both directions. A change can be backward-compatible (new code reads old data fine) but forward-incompatible (old code crashes reading new data with the new column).
Lesson 05 / 10Streaming

Structured Streaming & Watermarks

Deduplication, stream-static joins, windowed rules engines, and exactly-once semantics — built from the fraud monitoring newsletter series.

The watermark concept
In streaming, events don't always arrive in order. A transaction that happened at 14:00 might arrive at 14:35 due to mobile network delays, Kafka lag, or upstream retries. The watermark defines how long Spark waits for late data before closing a time window and discarding its state.

df.withWatermark("event_time", "30 minutes") tells Spark: once the max event_time seen so far exceeds a window's end by 30 minutes, that window's state can be safely dropped. Events arriving after the watermark threshold are discarded.
📋
The calibration trade-off: Watermark too tight → late events discarded, incorrect aggregations. Watermark too loose → state accumulates indefinitely, OOM on executors. Profile your Kafka consumer lag and source timestamp distribution to find the right value. For fraud: 2 hours is common.
Deduplication in streaming
The naive approach — dropDuplicates(["event_id"]) without a watermark — maintains state for every event_id ever seen. On a high-throughput Kafka topic, this causes unbounded state growth and eventually OOMs the executor.

The correct production approach (Spark 3.5+):
df.withWatermark("event_time", "2 hours").dropDuplicatesWithinWatermark(["event_id"])

This only tracks event_ids within the watermark window. Once the watermark advances past an event_time, that window's deduplication state is safely released.
Window types
Window typeBehaviourUse case
TumblingNon-overlapping fixed intervals: [0:00–1:00), [1:00–2:00)Hourly aggregations, batch-style windowed stats
SlidingOverlapping intervals: every 30s for a 5-min windowRolling averages, real-time velocity checks
SessionGap-based: window closes after a period of inactivityUser session tracking, clickstream analysis
Exactly-once semantics (the real answer)
Saying "Spark handles exactly-once" is a mid-level answer. The senior answer specifies the mechanism:

1. Kafka source with startingOffsets and checkpointing — Spark records the last committed offset to a checkpoint location.
2. foreachBatch with an idempotent write pattern — each micro-batch writes to Delta using MERGE with (event_id, window_start) as the idempotency key.
3. Kafka consumer offset is committed only after the Delta MERGE succeeds atomically.

If the job crashes mid-batch, it restarts from the last committed offset and re-processes the same events — the MERGE is idempotent, so re-processing produces the same result. This gives end-to-end exactly-once semantics.

Knowledge check

What's wrong with dropDuplicates(["event_id"]) without a watermark? ⌄
Without a watermark, Spark must maintain a deduplication state entry for every unique event_id ever seen since the stream started. On a high-throughput topic receiving millions of events per day, this state grows without bound. Eventually executor memory is exhausted, the state store overflows, and the job crashes or slows to a crawl. Always pair dropDuplicates with withWatermark, or use dropDuplicatesWithinWatermark (Spark 3.5+) to bound state to the watermark window.
You need to detect users who make 10+ transactions in any rolling 5-minute window. What window type and trigger? ⌄
Window type: Sliding window — window("event_time", "5 minutes", "30 seconds"). A tumbling window would miss transactions that span the boundary. A sliding window with 30-second slide gives near-real-time detection with manageable state overlap.

Trigger: processingTime='10 seconds' for sub-30-second latency while maintaining reasonable throughput. Pair with withWatermark("event_time", "30 seconds") to bound state. Alert when count(*) > 10 in any window.
Lesson 06 / 10Data Quality

6 Data Quality Checks for Every Pipeline

The silent failures that broke executive dashboards for 5 days — and the checks that would have caught them in minutes.

Why silent failures are worse than job failures
A failed job triggers an alert in your monitoring system immediately. An on-call engineer gets paged, investigates, and fixes it. Data consumers see an error and know to wait.

A silent failure is when a pipeline runs successfully, writes data, but writes wrong data. No alert fires. The dashboard shows numbers. Executives make decisions based on those numbers. Days later, someone notices the revenue figures are 30% off. Now you're investigating root cause across a week of pipeline runs, rebuilding trust, and potentially reprocessing a week of data.
📋
The fail-fast principle: Build quality assertions before the write, not after. If the data doesn't pass quality checks, fail the job loudly. A failed job is recoverable; silently wrong data that propagates downstream is not.
The 6 essential checks
1

Null rate on critical columns

Define acceptable null rates per column (e.g. customer_id: 0%, email: <5%). Alert or fail if exceeded.

2

Uniqueness / deduplication

For tables with expected unique keys, assert df.count() == df.dropDuplicates(["order_id"]).count(). Unexpected duplicates indicate upstream re-sends or join explosions.

3

Referential integrity

Verify that foreign keys in fact tables have matching records in dimension tables. Orphaned fact rows indicate upstream pipeline issues and break star schema joins silently.

4

Range bounds plausibility

Check that numeric columns are within expected ranges (e.g. amount > 0, age between 0 and 120). Extreme outliers from upstream bugs can skew aggregations dramatically.

5

Schema conformance

Verify column types and nullability match the expected schema before writing. Type mismatches (e.g. amount arriving as string) can either cause write failures (good) or silent type coercion (dangerous).

6

Row-count delta between runs

Track current_run_count / previous_run_count. Alert if the ratio is outside a threshold (e.g. ±20% from the 7-day rolling average). This is the simplest proxy for completeness failures.

Knowledge check

Your pipeline wrote 1.2M rows yesterday and 300K today. The job succeeded. What happened and how do you catch it? ⌄
This is a completeness failure — a 75% row-count drop that the job itself didn't catch because the writes succeeded (just with less data). Root causes: a missed upstream Kafka partition, a source table refresh failure, a date filter bug writing only one day's data instead of the full backfill.

Catch it with: The row-count delta check — alert if today_count / 7day_avg falls below 0.7 (30% drop threshold). This fires before the data reaches dashboards. Then investigate: check upstream source counts, review the job logs for partition skipping, verify the date filter in the pipeline code.

Compute Selection Guide

Why scheduled jobs on All-Purpose clusters bleed thousands per month — and how to fix it in 15 minutes.

The four compute types

⚠
The most common cost mistake: Running scheduled nightly jobs on All-Purpose clusters. These jobs spin up at 2am, run for 45 minutes, and then the cluster sits idle until morning — billing DBUs the whole time. Moving to Jobs clusters is a 15-minute fix that typically reduces ETL compute cost by 60-70%.
5 cost reduction strategies
1
Migrate scheduled jobs to Jobs clusters — single biggest win, typically 60-70% cost reduction for batch workloads. Takes 15 minutes in the Workflows UI.
2
Spot/preemptible instances for batch ETL — 60-80% cheaper than on-demand. Configure on-demand fallback for the driver node and set a max spot bid. Occasional interruptions are handled by Databricks' retry logic.
3
SQL Warehouse auto-suspend — set auto-suspend to 5-10 minutes. A warehouse left running overnight costs as much as a full day of queries.
4
Cluster policies — define max_workers caps per team in the admin console. Prevents runaway large clusters created by engineers who "just needed a quick test".
5
Cost attribution via system.billing.usage — tag all clusters by team and pipeline. Query system.billing.usage to identify the top 5 cost consumers each week.

Knowledge check

Your BI team's SQL Warehouse runs 24/7 and barely gets queries after 6pm. What do you do? ⌄
Enable auto-suspend with a 10-minute idle timeout. The warehouse will automatically suspend when no queries have run for 10 minutes, resuming in seconds when the next query arrives. For a warehouse unused from 6pm to 9am (15 hours), this eliminates 15 hours × DBU rate of wasted spend — typically thousands per month for a Medium or Large warehouse.
Lesson 08 / 10DebuggingSpark

The Debugging Maturity Ladder

How engineers progress from trial-and-error to finding the one bottleneck that matters — and the 7-question framework that exposes where you are.

The maturity ladder
Junior
Trial and error: Tweaks random configs hoping it gets faster. Increases shuffle.partitions without knowing if shuffles are even the bottleneck. Re-runs the job after each change with no systematic understanding of what changed.
Mid
Analysis paralysis: Reads every Spark optimisation doc, understands the techniques, but can't prioritise which one to apply to this specific job. Implements three improvements simultaneously and can't tell which one helped.
Senior
Diagnostic framework first: Opens Spark UI, finds the longest stage, checks task distribution for skew, looks at the SQL plan tab for missing pushdown — before touching any config. Identifies the single bottleneck, fixes it, and measures the improvement.
Principal
Proactive instrumentation: Adds stage-level metrics, data quality alerts, and SLA tracking before pipelines go to production. Owns the SLA rather than reacting to breaches.
The 7-question diagnostic chain
1

Spark UI → Stages tab: Which stage is longest?

90% of runtimes live in 1-2 stages. Find it before doing anything else.

2

Is the task duration distribution skewed?

In the stage detail, check if p99 task duration is 5×+ the median. If yes: data skew. If tasks are uniformly slow: different problem.

3

How large is the shuffle read/write?

Excessively large shuffle suggests missing predicate pushdown or a join that could be broadcast instead of shuffle-merge.

4

SQL tab: What join strategy is being used?

Is a SortMergeJoin running where a BroadcastHashJoin should? Are predicates shown as pushed down to the scan, or applied after reading all data?

5

What's the executor GC time?

If GC time exceeds 10% of task time, the cluster is memory-starved. Increase executor memory or reduce parallelism.

6

What's the actual per-partition size?

If average partition is >512 MB, increase shuffle.partitions. If average is <1 MB with thousands of tiny tasks, decrease it. Target 64-256 MB per partition.

7

Is data drift causing the slowdown?

Did the data volume change? New join keys with higher cardinality? A 3× slowdown that started on a specific date often means upstream data characteristics changed, not Spark config.

Knowledge check

A pipeline runs 6 hours against a 2-hour SLA. Your first action is to increase spark.sql.shuffle.partitions from 200 to 2000. Is this right? ⌄
No — this is the junior/mid-level trap. Changing shuffle.partitions without first diagnosing the bottleneck may make things worse or have no effect at all.

The correct first action: open Spark UI → Stages tab → find the longest stage. Then check task distribution. If the longest stage has 200 uniformly slow tasks (not skewed), increasing partitions may help. But if the stage has 5 tasks that ran for 5 hours and 195 that finished in 2 minutes, the problem is data skew — more partitions won't fix that, you need AQE skew handling or salting on the join key.
Lesson 09 / 10ModellingArchitecture

Why 3NF Is Killing Your Dashboards

Stop normalising analytical tables. The textbook was written for a different database.

Why 3NF was designed for a different world
Third Normal Form (3NF) was designed for OLTP row-store databases (MySQL, PostgreSQL) in the 1970s. Its goals: minimise storage redundancy, maximise write consistency, avoid update anomalies. It achieves this by breaking data into many small, normalised tables with foreign key relationships.

Databricks stores data as columnar Parquet files, not row stores. Columnar storage already eliminates redundancy at the physical level (column encoding + compression). Joining 8 normalised tables = 7 shuffle stages = expensive at petabyte scale. The 3NF trade-off — storage vs compute — flips completely in a lakehouse.
The lakehouse architecture pattern
Bronze (Raw)
  • Exact copy of source data, no transformation
  • Schema-on-read, often JSON or Parquet
  • Append-only — never overwrite
  • Partitioned by ingestion_date for cheap time-travel
  • Audit trail for compliance and debugging
Silver (Curated)
  • Cleaned, typed, deduplicated
  • Can be 3NF here for governance — normalised relationships
  • Schema enforcement enforced
  • Where data quality checks live
  • Single source of truth for domain entities
Gold (Aggregated / Serving): Wide, denormalised tables pre-joined for specific analytical use cases. A Gold table for the "Sales Dashboard" might join customer, order, product, and campaign tables into one wide fact table — so dashboard queries read a single table with a single predicate pushdown rather than joining 4 tables with 3 shuffles. Storage is cheap; the join computation is not.
SCD Type 2 with Delta MERGE
Slowly Changing Dimensions Type 2 (track historical values with effective dates) is implemented in Delta using MERGE:

Pass 1: UPDATE existing rows where the key matches and the value has changed — set is_current = false, effective_to = current_timestamp().
Pass 2: INSERT new rows for changed or new keys — set is_current = true, effective_from = current_timestamp(), effective_to = NULL.

At 500M rows, use hash comparison in the WHEN MATCHED clause: WHEN MATCHED AND source.row_hash != target.row_hash THEN UPDATE. This skips unchanged rows without column-by-column comparison.

Knowledge check

Why is a wide denormalised Gold table faster than a normalised Silver table for dashboard queries? ⌄
Three reasons:

1. No shuffles: Joining 4 tables requires 3 shuffle stages in Spark. A single wide table is a single scan — zero shuffles. At PB scale, shuffle is the dominant cost.

2. Better predicate pushdown: A single-table scan can push predicates directly to Parquet file min/max stats. Multi-table joins must evaluate predicates after the join, forcing full scans of all tables.

3. Better column encoding: Wide tables allow Parquet to apply per-column compression across all rows. Columnar compression is most effective on columns with low cardinality (status, region) — which appear repeatedly in a denormalised wide table.
Lesson 10 / 10CareerStrategy

Interview Strategy: From Correct to $200k

Why two engineers with correct answers get different salary bands — and the signals that put you in the top tier.

Why two correct answers get different offers
The newsletter's core observation from 200+ interviews: being technically correct is table stakes at the senior level. Everyone interviewing for a $180-220k role can answer "what is Liquid Clustering." The differentiation happens in the follow-up questions:

Mid-level answer: Picks a technology and explains it correctly.
Senior answer: Asks clarifying questions about the use case, explains the decision framework, describes the trade-offs and costs, mentions what could go wrong.

The 3-round interview breakdown shows that Round 1 (technical screen) is passed equally by both. Round 2 (system design) and Round 3 (hiring manager) are where the gap opens — because system design and hiring manager rounds test thinking patterns, not knowledge recall.
3 signals that separate $180k from $220k
Signal 1
Diagnostic before prescription. When asked "how would you fix this slow pipeline?", the senior engineer says "First, tell me what the Spark UI shows in the stages tab" rather than immediately listing fixes. The interviewer hears: this person won't make changes blindly.
Signal 2

Knowledge check

An interviewer asks: "Should we use Liquid Clustering or partitioning for our new 3 TB table?" What's the senior response? ⌄
The senior response starts with questions, not an answer:

"It depends on your query patterns. Can you tell me — what column do analysts most commonly filter on? How many distinct values does it have? Do you expect the access patterns to change as the product evolves? And what's the current query latency SLA?"

After hearing the answers: "Given that you filter primarily by customer_id (5M distinct values) and expect to add new filter dimensions as the product grows, I'd recommend Liquid Clustering by (customer_id, date). The high cardinality and evolving patterns are exactly where LC outperforms static partitioning. The trade-off is that existing data isn't immediately re-clustered — you'd run OPTIMIZE on a schedule. For a new 3 TB table, there's no migration cost."

The structure: clarify → framework → recommendation → trade-offs → validation. That's the $220k answer pattern.

Storage optimisation

Liquid Clustering vs Partitioning + Z-ORDER

  • Partitioning wins with stable, low-cardinality filter columns (<1,000 distinct values)
  • Liquid Clustering wins when filter columns change, or cardinality is high (5M+ distinct values)
  • Mutually exclusive — cannot add CLUSTER BY to a table with partition columns
  • Migrating a 2 TB table = full rewrite: compute cost + downtime + downstream schema changes
  • Z-ORDER also mutually exclusive with CLUSTER BY — both use Hilbert space-filling curves
  • LC adds a tree-based balancer on top: handles skew and cardinality automatically
  • New tables → default LC. Existing stable partitioned tables → leave them
Spark internals

SQL → Tasks: Catalyst + Tungsten pipeline

  • Parse → Analysis → Logical optimisation → Physical planning → Code gen → Tasks
  • Catalyst: constant folding, predicate pushdown, column pruning, join reordering
  • AQE: runtime partition coalescing, skew join splitting, join strategy switching
  • Tungsten: whole-stage code gen compiles plan to JVM bytecode — eliminates virtual dispatch
  • Partition pruning (directory-level) vs data skipping (file/row-level min/max stats)
  • Listing optimisation techniques = mid-level. Diagnostic framework first = senior
Delta Lake

Schema evolution & ACID guarantees

  • Default: strict enforcement — fails on any schema drift
  • mergeSchema=true: allows column additions only (not renames or drops)
  • Renames and type changes require explicit DDL: ALTER TABLE … RENAME COLUMN
  • Senior answer: discuss backward/forward compatibility impact on downstream consumers
  • ACID via optimistic concurrency + write-ahead log (_delta_log)
  • Time travel via versioned Parquet files — VACUUM controls retention (default 7 days)
Data quality

6 essential pipeline checks

  • Null rate on critical columns
  • Uniqueness / deduplication
  • Referential integrity (FK lookup against dimension tables)
  • Range bounds plausibility (min/max sanity)
  • Schema conformance (types, nullability)
  • Row-count delta between runs — simplest freshness/completeness proxy
  • Silent failures (wrong data, no alert) are more dangerous than job failures
Streaming

Structured Streaming & watermarks

  • withWatermark() defines late-data tolerance; state is dropped beyond the threshold
  • dropDuplicatesWithinWatermark (Spark 3.5+): dedup within watermark window
  • Stream-static join: broadcast static Delta table into executor memory
  • Tumbling: non-overlapping fixed intervals. Session: gap-based grouping
  • Trigger.AvailableNow for catch-up batches; Trigger.Continuous for sub-second latency
  • Watermark too tight = missed late events; too loose = unbounded state memory
Compute & cost

Cluster type selection

  • All-Purpose: interactive notebooks — always-on, ~3× the DBU cost of Jobs clusters
  • Jobs clusters: ephemeral, spin up per run — use for all scheduled ETL
  • SQL Warehouses: BI/SQL, auto-suspend, per-query billing
  • Serverless: fastest cold start, no cluster management, premium price
  • Biggest waste: scheduled jobs running on All-Purpose clusters
  • Tag clusters by team/pipeline for cost attribution in system.billing.usage
Modelling

Why not 3NF in Databricks

  • 3NF minimises storage for OLTP row stores — wrong trade-off for columnar Parquet
  • Joins across 8+ normalised tables = multiple shuffle stages = expensive at PB scale
  • Lakehouse: bronze (raw) → silver (cleansed, 3NF for governance) → gold (wide, denormalised)
  • Wide star schema reduces join count; storage is cheap, CPU is not
  • Predicate pushdown is most effective on wide tables with few joins
Debugging

Maturity ladder: junior → principal

  • Junior: trial and error — tweaks random configs
  • Mid: analysis paralysis — reads every doc, can't prioritise the bottleneck
  • Senior: diagnostic first — reads Spark UI, identifies the one bottleneck that matters
  • Principal: proactive — instruments pipelines before they fail; owns the SLA
  • Key signal: can you explain WHY a pipeline is slow before touching any config?

Decision tree — Liquid Clustering vs Partitioning (for 2TB+ tables)

Q1Are filter columns stable and low-cardinality (<1,000 distinct values)? Do all queries filter on the same column?
YES→ Keep partitioning. Add Z-ORDER for a secondary filter within partitions. Migration cost exceeds gain for stable tables.
NO→ Proceed to Q2.
Q2Do filter columns change over time, or is cardinality high (>100k distinct values)?
YES→ Use Liquid Clustering. CLUSTER BY the columns analysts actually query. Re-key in future with ALTER TABLE — no full rewrite needed.
NO→ Do downstream consumers depend on the physical file path layout (e.g. Hive-partitioned readers)? If yes, keep partitioning. Otherwise evaluate LC.
