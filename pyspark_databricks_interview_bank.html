Q1
What is PySpark and how does it differ from Pandas?
BasicsEasy
▾
Answer

PySpark is the Python API for Apache Spark — a distributed computing framework that processes data across a cluster of machines. Unlike Pandas (single-machine, in-memory), PySpark splits data into partitions spread across executor nodes, enabling petabyte-scale processing.

  • Pandas: in-memory, single machine, mutable DataFrames, eager execution
  • PySpark: distributed, immutable DataFrames, lazy evaluation, Catalyst optimizer

PySpark DataFrames are fault-tolerant via lineage (DAG) — if a partition fails, Spark recomputes it from source transformations.

Interview Tip

Mention lazy evaluation unprompted — it shows depth. "Transformations build a logical plan; execution happens only when an action like count() or write() is triggered."

Q2
What is lazy evaluation and why does Spark use it?
BasicsEasy
▾
Answer

Transformations (filter, join, withColumn) don't execute immediately — they build a logical plan (DAG). Execution triggers only on an action (count(), show(), write()).

Why: Spark's Catalyst optimizer can then inspect the full plan, push filters down to the scan, eliminate unused columns, reorder joins, and fuse stages — producing a much more efficient physical plan than executing eagerly step-by-step.

Interview Tip

Follow-up is often: "How do you debug a slow pipeline?" — Answer: use explain(True) to print the physical plan and identify missing predicate pushdowns or unexpected shuffles.

Q3
RDD vs DataFrame vs Dataset — what are the differences?
BasicsEasy
▾
Answer
  • RDD: Low-level, untyped, no schema, no Catalyst optimization, supports any data type. Use only when you need fine-grained control over partitioning/serialization.
  • DataFrame: Schema-aware, distributed table. Catalyst optimizer + Tungsten execution. Best performance for structured data in Python/Scala/Java/R.
  • Dataset: Typed DataFrame (Scala/Java only). Compile-time type safety. Not available in PySpark.

In PySpark, always prefer DataFrame API. RDDs are used only for legacy code or custom transformations Spark SQL doesn't support.

Q4
What is the difference between repartition() and coalesce()?
BasicsMedium
▾
Answer
  • repartition(n): Wide transformation — full shuffle across the cluster. Can increase OR decrease partitions. Produces evenly balanced partitions. Use before a heavy groupBy/join on skewed data.
  • coalesce(n): Narrow transformation — no full shuffle. Can only decrease partitions by merging adjacent ones. Faster, but can produce uneven partitions. Use before writing to reduce output files.
# 200 → 8 partitions before write (narrow, fast)
df.coalesce(8).write.parquet(output_path)

# Re-shuffle to fix skew before join
df.repartition(200, col("customer_id")).join(...)
Interview Tip

Key distinction: coalesce can't increase partitions. If you call coalesce(500) on a 200-partition DF, you still get 200. Use repartition to increase.

Q5
Narrow vs Wide transformations — explain with examples
BasicsMedium
▾
Answer
  • Narrow: Each input partition maps to exactly one output partition. No data moves across the network. Examples: filter(), map(), withColumn(), select(), coalesce().
  • Wide: Data from multiple input partitions is required to compute each output partition. Causes a shuffle (network I/O). Examples: groupBy(), join(), repartition(), distinct(), orderBy().

Shuffles are the main performance cost in Spark. Minimise them by filtering early, broadcasting small tables, and using partition-aware joins.

Q6
What is a Broadcast Join and when should you use it?
PerformanceMedium
▾
Answer

A broadcast join sends the smaller DataFrame to every executor node so each executor can join locally without shuffling the large DataFrame. Eliminates the most expensive operation in a standard sort-merge join.

from pyspark.sql.functions import broadcast

# dim_products is small (< 10MB) — broadcast it
result = fact_orders.join(
    broadcast(dim_products),
    "product_id", "inner"
)

When to use: Right-side table < 10MB (default spark.sql.autoBroadcastJoinThreshold = 10MB). Spark auto-broadcasts below this threshold; use explicit hint to force it.

Interview Tip

Mention the config: spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 50*1024*1024) to raise the threshold to 50MB. Or -1 to disable auto-broadcast.

Q7
How do you handle data skew in PySpark?
PerformanceMedium
▾
Answer

Data skew = one partition holds far more data than others → that task runs long while others idle (straggler problem). Solutions:

  • Salting: Append a random suffix to the join key to spread one hot key across multiple partitions, then aggregate back.
  • Broadcast join: If the smaller side is small enough, broadcast it and skip the shuffle entirely.
  • Repartition by column: df.repartition(200, col("key")) distributes by hash more evenly.
  • AQE (Adaptive Query Execution): Spark 3+ auto-detects skew at runtime and splits large partitions. Enable with spark.sql.adaptive.enabled = true.
# Salting example
import pyspark.sql.functions as F
n_salt = 20
df_salted = df.withColumn("salt", (F.rand() * n_salt).cast("int"))
df_salted = df_salted.withColumn("skewed_key_salted",
    F.concat(col("skewed_key"), F.lit("_"), col("salt")))
Q8
cache() vs persist() — what's the difference?
PerformanceAdvanced
▾
Answer

cache() is shorthand for persist(StorageLevel.MEMORY_AND_DISK) — stores in memory, spills to disk if needed.

persist() gives full control over storage level:

  • MEMORY_ONLY — fastest, but evicts if memory is full (recomputes from lineage)
  • MEMORY_AND_DISK — spills overflow to disk
  • DISK_ONLY — for very large DFs that don't fit in memory
  • MEMORY_ONLY_SER — serialized format, more GC-friendly but slower to read
df.persist(StorageLevel.MEMORY_AND_DISK)
df.count() # trigger materialisation
# ... reuse df many times ...
df.unpersist() # release when done
Interview Tip

Always call an action after persist() to actually materialise it. And always call unpersist() when done — Spark doesn't auto-release cached data promptly.

Q9
How does Adaptive Query Execution (AQE) work in Spark 3?
AdvancedHard
▾
Answer

AQE re-optimises query plans at runtime using actual intermediate statistics, rather than relying only on pre-execution estimates. Three key features:

  • Dynamic partition coalescing: After a shuffle, merges small post-shuffle partitions to reduce task overhead.
  • Dynamic skew handling: Detects skewed shuffle partitions and splits them into smaller tasks automatically.
  • Dynamic join strategy switching: Can convert a sort-merge join to a broadcast join if the build side turns out to be small after filtering.
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

AQE is enabled by default in Databricks Runtime 7.3+.

Q11
Explain the Catalyst Optimizer and Tungsten execution engine
AdvancedHard
▾
Answer

Catalyst Optimizer — Spark SQL's query optimiser. Passes through 4 phases:

  • Analysis: Resolves column names and types against the catalog.
  • Logical Optimization: Rule-based rewrites — predicate pushdown, constant folding, column pruning.
  • Physical Planning: Generates multiple physical plans (e.g., broadcast vs sort-merge join), selects the cheapest via cost model.
  • Code Generation: Tungsten generates bytecode (whole-stage codegen) that runs at near-native speed.

Tungsten — Memory and CPU efficiency layer: off-heap memory management, cache-aware computation, SIMD-style vectorised operations, and whole-stage code generation to eliminate virtual function dispatch overhead.

Interview Tip

Use df.explain(True) to print all four Catalyst plans: Parsed → Analyzed → Optimized Logical → Physical. Interviewers love candidates who know this.

Q12
distinct() vs dropDuplicates() — what is the difference?
BasicsMedium
▾
Answer
  • distinct() — considers all selected columns. If you've selected 3 columns, all 3 must match for a row to be considered a duplicate.
  • dropDuplicates(['col1', 'col2']) — deduplicates based only on the specified subset of columns. More flexible: keeps one row per unique combination of the named keys.
# All columns must match
df.select("customer_id", "email").distinct()

# Only customer_id determines uniqueness (keeps first by default)
df.dropDuplicates(["customer_id"])

For SCD Type 2 deduplication on business keys, always use dropDuplicates(['key1', 'key2']).

Find the highest salary employee in each department
WindowMediumRanking
▾
Answer — row_number() over partition, filter == 1
from pyspark.sql import Window
from pyspark.sql.functions import row_number, col

w = Window.partitionBy("department").orderBy(col("salary").desc())

result = (df
    .withColumn("rn", row_number().over(w))
    .filter(col("rn") == 1)
    .select("department", "employee_id", "salary")
)
Output

Interview Tip

Use row_number() here, not rank(). If two people tie for top salary, rank() returns both — which may not be desired. row_number() always returns exactly 1 winner (arbitrary tie-break).

W3
rank() vs dense_rank() vs row_number() — explain with example
WindowMediumRanking
▾
Answer

For salaries 1000, 1000, 800 within a partition:

row_number: 1, 2, 3   # always unique, arbitrary tie-break
rank:       1, 1, 3   # ties share rank, next rank skipped (gap)
dense_rank: 1, 1, 2   # ties share rank, NO gap
from pyspark.sql.functions import rank, dense_rank, row_number

w = Window.partitionBy("dept").orderBy(col("salary").desc())
df.withColumn("rn",         row_number().over(w)) \
  .withColumn("rank",       rank().over(w)) \
  .withColumn("dense_rank", dense_rank().over(w))
Interview Tip

Use dense_rank for "second highest salary" queries — it guarantees the next distinct salary is always rank 2, regardless of how many people share rank 1.

Calculate month-over-month revenue growth percentage

▾

W8
Calculate a running total (cumulative sum) of sales
WindowMediumRunning
▾
Answer
from pyspark.sql.functions import sum as _sum, col
from pyspark.sql import Window

w = Window.orderBy(col("date")) \
    .rowsBetween(Window.unboundedPreceding, Window.currentRow)

result = df.withColumn("running_total", _sum(col("sales")).over(w))
Frame Syntax

rowsBetween(unboundedPreceding, currentRow) = "from the start of the partition to the current row." This is the correct frame for running totals. Without rowsBetween, the default frame is rangeBetween(unboundedPreceding, currentRow) which can include future rows with the same order value.

Explain all 6 join types with use cases
JoinMedium
▾

All 6 PySpark join types and when to use them:

# Inner  — only matching rows on both sides
orders.join(products, "product_id", "inner")

# Left   — all orders, nulls if no product match
orders.join(products, "product_id", "left")

# Right  — all products, nulls if no order match
orders.join(products, "product_id", "right")

# Full   — all rows from both, nulls where no match
orders.join(products, "product_id", "full")

# Left Anti  — orders with NO matching product (orphans)
orders.join(products, "product_id", "left_anti")

# Left Semi  — orders WHERE product exists (no product cols)
orders.join(products, "product_id", "left_semi")
Most Tested

left_anti is a favourite: "Find orders referencing products that don't exist in the product master." It returns left rows with NO match in the right DF, and includes NO right-side columns.

DatabricksAdvancedHard
▾

from delta.tables import DeltaTable

delta_tbl = DeltaTable.forName(spark, "catalog.schema.customers")

# Step 1: Expire changed records

    incoming.alias("src"),
    "tgt.customer_id = src.customer_id AND tgt.is_current = true"

        .saveAsTable("catalog.schema.customers")
Databricks-Specific Questions
Delta Lake, Unity Catalog, Auto Loader, Structured Streaming, Medallion — Databricks interview essentials
D1
What is Delta Lake and what are its key features?
Delta LakeMedium
▾
Answer

Delta Lake is an open-source storage layer that brings ACID transactions to Apache Spark and big data workloads. It stores data as Parquet files with a transaction log (_delta_log/) that records every change.

  • ACID transactions: Serialisable isolation via the transaction log. No partial writes.
  • Time Travel: Query data as it was at any previous version or timestamp.
  • Schema enforcement & evolution: Rejects writes with incompatible schema; allows evolution with mergeSchema.
  • Upserts via MERGE: Efficient upsert/delete operations without full rewrites.
  • Scalable metadata: Transaction log avoids full directory listings for large tables.
  • Auto-optimise: Compacts small files and writes optimised files automatically.
D2
Explain VACUUM, OPTIMIZE, and Z-ORDER — and when to use each
Delta LakeMedium
▾
Answer
  • VACUUM: Removes data files no longer referenced by the transaction log (and older than the retention threshold, default 7 days). Reclaims storage but breaks time travel beyond the retention window.
  • OPTIMIZE: Compacts many small files into larger ones (target ~1GB). Reduces file-open overhead on reads. Does NOT reorder data.
  • Z-ORDER BY: Co-locates related data in the same files using multi-dimensional clustering. Dramatically improves query performance when filtering on z-ordered columns. Typically combined with OPTIMIZE.
-- Compact small files
OPTIMIZE catalog.schema.orders;

-- Compact + co-locate data for common filter columns
OPTIMIZE catalog.schema.orders
ZORDER BY (customer_id, order_date);

-- Remove stale files older than 7 days
VACUUM catalog.schema.orders RETAIN 168 HOURS;
Interview Tip

Z-ORDER is best on high-cardinality columns you filter on frequently (customer_id, device_id). It degrades with too many columns — keep it to 1-3 per table. For append-mostly tables, prefer Liquid Clustering (Databricks Runtime 13.3+) — it's incremental and doesn't require full rewrites.

▾
Answer

Deletion Vectors (DVs) are a performance optimisation for DELETE and MERGE operations on Delta Lake tables. Instead of rewriting entire Parquet files when rows are deleted/updated, Spark writes a small sidecar file (.dv) that records which row IDs in the original file are logically deleted.

  • Reads filter out DV-marked rows on the fly.
  • DML operations (DELETE, UPDATE, MERGE) become much faster — no file rewrite needed immediately.
  • DVs are cleaned up during OPTIMIZE, which rewrites affected files at that point.
-- Enable deletion vectors
ALTER TABLE catalog.schema.orders
SET TBLPROPERTIES ('delta.enableDeletionVectors' = 'true');

What is Medallion Architecture (Bronze → Silver → Gold)?
DatabricksMedium
▾
Answer
  • Bronze (Raw): Raw ingestion layer. Data arrives exactly as-is from source. No transformations. Schema-on-read. Append-only. Acts as the immutable source of truth for replay.
  • Silver (Cleaned): Deduplicated, validated, type-cast, PII-masked, schema-enforced. One row per business key per event. Quality-checked with DQ rules. Serves as the canonical cleansed dataset.
  • Gold (Aggregated): Business-level aggregates, KPIs, denormalised fact+dimension tables. Optimised for consumption by BI tools, ML models, and downstream APIs.
Production Pattern

In each layer, partition tables on date columns and Z-ORDER on filter columns. Write to Unity Catalog managed tables. Use Auto Loader for Bronze ingestion, DeltaTable MERGE for Silver upserts, and scheduled Gold refresh jobs.

What is Auto Loader and why use it over spark.read?
DatabricksMedium
▾
Answer

Auto Loader (cloudFiles) is Databricks' incremental ingestion framework. It continuously monitors a cloud storage path for new files and processes only new arrivals — without re-scanning the entire directory.

  • File notification mode: Uses cloud events (S3 SQS / Azure Event Grid) for near-real-time file discovery. Scales to millions of files without directory listing.
  • Directory listing mode: Periodic scan, simpler setup.
  • Schema inference + evolution: Infers schema from incoming files and handles new columns with rescuedDataColumn.
df = spark.readStream.format("cloudFiles") \
    .option("cloudFiles.format", "json") \
    .option("cloudFiles.schemaLocation", schema_path) \
    .load(source_path)

df.writeStream.format("delta") \
    .option("checkpointLocation", checkpoint_path) \
    .trigger(availableNow=True) \
    .toTable("catalog.bronze.events")

Structured Streaming — watermarks and late-arriving data
DatabricksAdvancedHard
▾
Answer

A watermark tells Spark how long to wait for late-arriving events before closing a time window. withWatermark("event_time", "10 minutes") means: tolerate data up to 10 minutes late. After the threshold, the window state is dropped.

df.withWatermark("event_time", "10 minutes") \
  .groupBy(
      window(col("event_time"), "5 minutes"),
      col("device_id")
  ) \
  .agg(_sum("value").alias("total"))
Production Pattern

For a two-band read strategy: process the latest window normally; use a LOOKBACK_HOURS window to catch late data from prior periods and MERGE it into the target Delta table rather than reprocessing from scratch.

What is FSCK REPAIR TABLE and when do you need it?
Delta LakeHard
▾
Answer

FSCK REPAIR TABLE fixes inconsistencies between a Delta table's transaction log and the actual files on storage. This can occur when files are deleted externally (not through Delta operations) or when storage-level corruption occurs.

  • Without REPAIR: queries fail with "file not found" errors even though the transaction log references those files.
  • With DRY RUN: shows what would be fixed without making changes.
FSCK REPAIR TABLE catalog.schema.events;

-- Preview only, no changes
FSCK REPAIR TABLE catalog.schema.events DRY RUN;
D10
Deep clone vs shallow clone in Delta Lake
Delta LakeHard
▾
Answer
  • Deep Clone: Copies all data files + transaction log to the destination. Fully independent — changes to source don't affect clone. Use for backups, cross-catalog migration.
  • Shallow Clone: Copies only the transaction log, references the original data files. Much faster and cheap, but dependent on the source — if source files are vacuumed, clone breaks. Use for dev/test environments.
-- Deep clone (full copy)
CREATE TABLE catalog_dev.schema.orders
DEEP CLONE catalog_prod.schema.orders;

-- Shallow clone (metadata only)
CREATE TABLE catalog_dev.schema.orders
SHALLOW CLONE catalog_prod.schema.orders;

What are Databricks Asset Bundles (DABs) and how do they enable CI/CD?
DatabricksAdvancedHard

Answer

DABs are Databricks' native IaC framework — define jobs, pipelines, clusters, and permissions in YAML (databricks.yml), version-control them in Git, and deploy via CLI or CI/CD pipelines.

  • databricks bundle deploy --target prod → deploys all resources to the target workspace.
