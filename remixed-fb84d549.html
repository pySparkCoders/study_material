Key Optimization Rules
Predicate Pushdown
Moves filter() / WHERE conditions toward data source. Reads only matching rows — skips entire Parquet row groups.
Projection Pruning
Removes unneeded columns from scan. Less I/O, especially with columnar formats like Parquet.
Constant Folding
Precomputes literal expressions at plan time. 10 * 2 becomes 20 before execution.
Combine Filters
.filter(A).filter(B) merged into a single filter(A AND B), avoiding redundant passes.
Join Reordering
Joins smaller tables first to minimize intermediate result sizes across multi-table queries.
Eliminate No-Ops
Removes redundant operations like double selects on the same columns.

Partitioning & Storage
Data Layout Optimization
repartition(n) vs coalesce(n)
repartition: full shuffle, can increase or decrease, even distribution. coalesce: no shuffle, only decreases, merges adjacent partitions. Use repartition before joins; coalesce after filtering.
partitionBy vs bucketBy
partitionBy: directory hierarchy, low cardinality columns, partition pruning. bucketBy: hash into N files, high cardinality columns, shuffle-free joins. bucketBy requires saveAsTable().
Real-Time Processing
Spark Streaming
Append Mode
Only new rows since last trigger written. Insert-only sinks.
Complete Mode
Full result table rewritten every trigger. Global aggregations.
Update Mode
Only changed rows written. Most efficient for streaming aggregations.

Infrastructure
AWS EC2 Machines as Databricks Clusters
Every Databricks classic cluster runs on EC2 instances inside your own AWS account
M-family (General)
Balanced 4GB RAM/vCPU. M5, M6i, M7i. Best for ETL, driver nodes, general workloads.
R-family (Memory)
8GB RAM/vCPU. R5, R6i, R7g. Best for large joins, caching, aggregations. Most common for Spark.
C-family (Compute)
2GB RAM/vCPU. C5, C6i, C7g. Best for CPU-bound batch transformations.
I-family (Storage)
NVMe local SSD. I3, I4i. Best for shuffle-heavy jobs, large spills to local disk.
P/G-family (GPU)
P3/P4d (V100/A100) for training. G5/G6 (A10G/L4) for inference. ML workloads.
Graviton (ARM)
M6g, R7g, C7g. Up to 40% cheaper than x86 equivalents. Requires DBR 9.1 LTS+.
Fleet Instances + Spot
r-fleet.*, m-fleet.* let Databricks pick across multiple EC2 families using Price Capacity Optimized allocation — dramatically reducing InsufficientInstanceCapacity errors on large Spot clusters. Use SPOT_WITH_FALLBACK for critical jobs. First executor On-Demand (driver), workers Spot.
Governance & Storage
Unity Catalog · S3 Volumes · IAM Auth
How Unity Catalog governs access to S3 through IAM roles, storage credentials, external locations, and volumes
Storage Credential
Wraps an AWS IAM Role ARN. Only identity allowed to access UC-managed data. Uses cross-account trust policy with ExternalId condition to prevent confused deputy attacks.
External Location
Combines a Storage Credential with an S3 path prefix. Defines the authorization scope. Multiple External Locations can share one Storage Credential.
Managed Tables/Volumes
UC owns lifecycle AND storage. DROP deletes S3 files after retention (8d tables, 7d volumes). Path chosen by UC automatically under __unitystorage/.
External Tables/Volumes
You own storage lifecycle. DROP removes only metadata — S3 files untouched. Must point to a path inside a registered External Location.
3-Level Namespace & Privilege Hierarchy
Metastore > Catalog > Schema > Table/Volume. Key grants: USE CATALOG, USE SCHEMA, SELECT, MODIFY on tables; READ VOLUME, WRITE VOLUME on volumes; CREATE EXTERNAL TABLE, CREATE EXTERNAL VOLUME on External Locations. Managed storage location priority: Schema overrides Catalog overrides Metastore.

Key Code Patterns
Salting — skewed join fix
from pyspark.sql.functions import col, lit, floor, rand, concat, explode, array

NUM_SALTS = 10
salted_large = large_df.withColumn("salted_key",
    concat(col("key"), lit("_"), floor(rand() * NUM_SALTS).cast("int")))

salted_small = small_df.withColumn("salt", explode(array([lit(i) for i in range(NUM_SALTS)])
).withColumn("salted_key", concat(col("key"), lit("_"), col("salt").cast("string")))

result = salted_large.join(salted_small, "salted_key").drop("salt", "salted_key")
Window functions — rank + running total
from pyspark.sql import Window
from pyspark.sql.functions import rank, avg, sum as _sum, col

w_rank = Window.partitionBy("dept").orderBy(col("salary").desc())
w_run  = Window.partitionBy("dept").orderBy("hire_date").rowsBetween(Window.unboundedPreceding, 0)

df.withColumn("sal_rank", rank().over(w_rank))
  .withColumn("running_total", _sum("salary").over(w_run))
