Delta Lake & Storage
Q1
What is Delta Lake and how does it differ from a plain Parquet data lake?
Easy▼

Delta Lake adds an ACID transaction layer on top of Parquet files via a JSON-based transaction log (_delta_log). Key differences:

  • ACID transactions — concurrent reads/writes are safe
  • Schema enforcement and evolution
  • Time travel — query older snapshots with VERSION AS OF or TIMESTAMP AS OF
  • MERGE, UPDATE, DELETE operations (not possible on plain Parquet)
  • Optimized compaction via OPTIMIZE and VACUUM
Hook to your work: "In my project, Delta Lake's MERGE was critical for SCD Type-2 updates on the Employee Dimension table — plain Parquet would have required full rewrites."
Q2
Explain VACUUM and OPTIMIZE. When would you run each, and what are the risks?
Medium▼

OPTIMIZE compacts small files into larger ones (default 1 GB target), improving read performance. Can include Z-ORDER to co-locate related data. Safe to run anytime — does not delete files.

VACUUM physically deletes files older than the retention threshold (default 7 days). Risk: if you VACUUM with a threshold lower than 7 days and a stream is reading from that table, you can break the stream by removing files it hasn't consumed yet. Also breaks time travel beyond the retained window.

PwC risk angle: In a consulting context, VACUUM must be governed. Never set retention below 7 days without explicit sign-off — audit and compliance teams need historical versions.
Q3
What is Liquid Clustering in Databricks, and when would you prefer it over Z-ORDER or partitioning?
Hard▼

Liquid Clustering (DBR 13.3+) is an incremental data layout optimization that replaces static partitioning and Z-ORDER. Key advantages:

  • Clustering columns can be changed without rewriting the table
  • Clustering happens incrementally — no need to rewrite the full table on OPTIMIZE
  • Avoids the "too many small partitions" problem caused by high-cardinality partition keys
  • Better for tables with multiple frequent filter columns

Prefer Liquid Clustering when: column cardinality is high, filter columns change over time, or you want to avoid partition skew. Prefer partitioning when downstream tools (e.g., ADF, ADLS file system) need folder-level access.

-- Enable liquid clustering
CREATE TABLE my_table
CLUSTER BY (event_date, region)
AS SELECT * FROM source;
Q4
How does the Delta transaction log handle concurrent write conflicts? What happens when two writers hit the same table simultaneously?
Brutal▼

Delta uses optimistic concurrency control. Each writer reads the current version N, performs its operation, then tries to commit as version N+1 by writing a new JSON entry to _delta_log/.

If two writers both read version N and both try to write N+1, the second commit fails with a ConcurrentModificationException. Delta then checks if the conflict is resolvable:

  • No overlap (e.g., writes touch different partitions) — Delta retries automatically
  • Overlapping files — the second writer must re-read and retry from scratch

Isolation levels: Delta supports Serializable (default, strongest) and WriteSerializable (weaker but allows more concurrency — concurrent appends are not treated as conflicting).

This matters for streaming + batch concurrent writes. Use delta.isolationLevel = WriteSerializable for append-only streaming tables to reduce conflicts.

Q9
SCD Type 1 vs SCD Type 2 — when do you use each? How is Type 2 implemented in Delta?
Medium▼

SCD Type-1: Overwrite — no history. Use when history doesn't matter (e.g., fixing a typo in a name field).

SCD Type-2: Add a new row with effective_start, effective_end, is_current flag. Full history preserved. Use for dimension tables where historical context matters (e.g., employee department history for ESG KPI reporting).

Delta MERGE pattern for SCD Type-2:

-- Step 1: Expire changed records
MERGE INTO dim_employee t
USING new_data s ON t.emp_id = s.emp_id AND t.is_current = true
WHEN MATCHED AND t.dept != s.dept THEN
  UPDATE SET t.effective_end = current_date(), t.is_current = false;

-- Step 2: Insert new/changed records
INSERT INTO dim_employee
SELECT *, current_date() as effective_start,
  date('9999-12-31') as effective_end, true as is_current
FROM new_data WHERE emp_id IN (changed_ids);
Your angle: "I added a first-run guard — on first load, I skip the expiry step entirely to avoid unnecessary MERGE overhead on empty target tables. And I filter incoming data by load_date watermark to avoid reprocessing historical batches."
Q10
Design a late-arriving data strategy for a Bronze pipeline ingesting files that can arrive up to 72 hours late. How do you ensure no data loss and no duplicate processing?
Brutal▼

Full strategy (your actual architecture):

  • Two-band read: Read T-72h to T-0 window on each run. Parameterize as LOOKBACK_HOURS = 72. Prevents missing late arrivals.
  • Idempotent writes: Use MERGE on natural key + batch_id so re-processing the same file doesn't create duplicates.
  • Watermark tracking: Store last_processed_watermark in a control Delta table. Each run reads from max(event_time) - LOOKBACK_HOURS.
  • File-level dedup: Auto Loader maintains a checkpoint of processed files. Even if a file is re-dropped, Auto Loader won't re-read it.
  • Structured streaming watermark: If using streaming, set .withWatermark("event_time", "72 hours") to bound state size while tolerating late data.
Gotcha: don't set streaming watermark too tight — if you set 24h but data arrives at 60h, those records are dropped silently. Validate against SLA requirements first.
Performance Tuning
Q11
When would you use a broadcast join vs a sort-merge join in PySpark?
Medium▼

Broadcast join: Use when one side of the join is small enough to fit in executor memory (default threshold: 10 MB, configurable via spark.sql.autoBroadcastJoinThreshold). Broadcasts the small table to all workers, avoiding shuffle entirely.

Sort-merge join: Used for large-to-large joins. Both datasets are shuffled, sorted on the join key, and merged. Expensive but handles any size.

from pyspark.sql.functions import broadcast

# Force broadcast
df_large.join(broadcast(df_small), "key")
Your project angle: "In my BOM hierarchy explosion pipeline joining 1 crore+ material keys, I identified the dimension lookup table as small enough to broadcast, which eliminated a multi-minute shuffle stage."
Q12
What is data skew and how do you handle it in a PySpark join?
Hard▼

Data skew happens when one partition receives significantly more data than others — one executor runs 10× longer while the rest finish, making the stage the bottleneck.

Detection: Check Spark UI stage metrics — if one task's runtime is 10× the median, skew is likely. Also check df.groupBy("key").count() for key distribution.

Fixes:

  • Salting: Add a random suffix (0 to N) to the skewed key in both DataFrames, join on salted key, then aggregate away the salt.
  • AQE Skew Join (Spark 3.x): Set spark.sql.adaptive.skewJoin.enabled = true — Databricks auto-splits skewed partitions.
  • Broadcast the skewed side if small enough.
# Salting example
from pyspark.sql.functions import col, concat, lit, rand, floor

n = 10
df_large = df_large.withColumn("salt", (floor(rand() * n)).cast("string"))
df_large = df_large.withColumn("salted_key", concat(col("key"), lit("_"), col("salt")))

df_small = df_small.crossJoin(spark.range(n).withColumnRenamed("id","salt_val"))
df_small = df_small.withColumn("salted_key", concat(col("key"), lit("_"), col("salt_val").cast("string")))

Q14
How does Auto Loader differ from COPY INTO for incremental data ingestion?
Hard▼
  • Auto Loader: Streaming-based, uses cloudFiles source. Tracks new files via cloud notification events (SNS/Event Grid) or directory listing. Stateful — remembers processed files in checkpoint. Best for continuously arriving files at high volume.
  • COPY INTO: Batch-based SQL command. Idempotent — re-running it won't re-load files already loaded. Simpler setup, no streaming infrastructure needed. Better for scheduled batch ingest of moderate file volume.

Key differences table:

  • Auto Loader scales to millions of files; COPY INTO degrades at very high file counts (full directory scan)
  • Auto Loader supports schema inference/evolution natively; COPY INTO requires explicit schema
  • Auto Loader is integrated with Structured Streaming — supports foreachBatch, watermarks; COPY INTO is SQL only
Your choice in production: "We used Auto Loader for all 5 ESG application feeds because of the high daily file volume and the need for schema evolution. COPY INTO was used only for one-time historical backfills."
Streaming & Structured Streaming
Q15
What is foreachBatch in Structured Streaming and why is it useful?
Medium▼

foreachBatch lets you apply any batch DataFrame operation to each micro-batch produced by a streaming query. This bridges the gap between streaming and batch APIs.

Common use cases:

  • MERGE operations into Delta (streaming MERGE is not natively supported — foreachBatch makes it possible)
  • Writing to multiple sinks from a single stream
  • Applying deduplication logic using joinWith or dropDuplicates
def upsert_batch(batch_df, batch_id):
    batch_df.createOrReplaceTempView("updates")
    batch_df._jdf.sparkSession().sql("""
        MERGE INTO silver_table t
        USING updates s ON t.id = s.id
        WHEN MATCHED THEN UPDATE SET *
        WHEN NOT MATCHED THEN INSERT *
    """)

stream.writeStream.foreachBatch(upsert_batch).start()
Q16
How does AQE (Adaptive Query Execution) work in Spark 3.x and what specific optimizations does it apply at runtime?
Brutal▼

AQE re-optimizes the query plan at runtime using shuffle stage statistics (actual row counts, data sizes). Enabled with spark.sql.adaptive.enabled = true (default in Spark 3.2+).

Three major runtime optimizations:

  • Dynamic coalescing of shuffle partitions: If spark.sql.shuffle.partitions = 200 but only 5 partitions have data after a filter, AQE merges them to avoid 195 nearly-empty tasks.
  • Dynamic switch to broadcast join: If the planner estimated a large join but actual data after filtering is small, AQE switches from sort-merge to broadcast join mid-execution.
  • Skew join optimization: Automatically splits skewed partitions and replicates the corresponding non-skewed side — no manual salting needed.

Key config:

spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "256MB")

P7
Implement a sessionization pipeline: given (user_id, event_time), group events into sessions where the gap between consecutive events exceeds 30 minutes.
Hard▼
from pyspark.sql.window import Window
from pyspark.sql.functions import (col, lag, unix_timestamp,
    when, sum as _sum, concat, lit)

w_user = Window.partitionBy("user_id").orderBy("event_time")

df = df.withColumn("prev_time", lag("event_time").over(w_user))

df = df.withColumn("gap_sec",
    unix_timestamp("event_time") - unix_timestamp("prev_time"))

df = df.withColumn("new_session",
    when((col("gap_sec") > 1800) | col("gap_sec").isNull(), 1).otherwise(0))

df = df.withColumn("session_num", _sum("new_session").over(
    w_user.rowsBetween(Window.unboundedPreceding, 0)))

df = df.withColumn("session_id",
    concat(col("user_id"), lit("_"), col("session_num")))

Cumulative sum of new_session flags gives a monotonically increasing session counter per user.

P10
Design an idempotent batch ingestion framework in PySpark. The same source file can arrive multiple times. Ensure exactly-once semantics without using streaming checkpoints.
Brutal▼
from pyspark.sql.functions import md5, col, lit, input_file_name, current_timestamp

# 1. Compute file fingerprint
df = spark.read.format("parquet").load(source_path) \
     .withColumn("source_file", input_file_name()) \
     .withColumn("batch_id", md5(lit(source_path))) \
     .withColumn("ingested_at", current_timestamp())

Find the second-highest salary in each department.
Easy▼
SELECT department, salary
FROM (
  SELECT department, salary,
         DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dr
  FROM employees
)
WHERE dr = 2;

Use DENSE_RANK rather than RANK so duplicate salaries at rank 1 don't skip to rank 3. Use DENSE_RANK rather than ROW_NUMBER to correctly handle ties at rank 2.

S3
Compute month-over-month revenue growth percentage for each region.
Medium▼
WITH monthly AS (
  SELECT region,
         DATE_FORMAT(order_date, 'yyyy-MM') AS month,
         SUM(revenue) AS monthly_rev
  FROM sales
  GROUP BY region, DATE_FORMAT(order_date, 'yyyy-MM')
)
SELECT region, month, monthly_rev,
       LAG(monthly_rev) OVER (PARTITION BY region ORDER BY month) AS prev_rev,
       ROUND(
         100.0 * (monthly_rev - LAG(monthly_rev) OVER (PARTITION BY region ORDER BY month))
         / NULLIF(LAG(monthly_rev) OVER (PARTITION BY region ORDER BY month), 0),
       2) AS mom_growth_pct
FROM monthly
ORDER BY region, month;

Use NULLIF(..., 0) to prevent division by zero for first month or months with zero prior revenue.

S5
Compute a 3-month rolling average revenue per region, ordered by month.
Medium▼
WITH monthly AS (
  SELECT region,
         DATE_FORMAT(order_date, 'yyyy-MM') AS month,
         SUM(revenue) AS monthly_rev
  FROM sales
  GROUP BY region, DATE_FORMAT(order_date, 'yyyy-MM')
)
SELECT region, month, monthly_rev,
       ROUND(AVG(monthly_rev) OVER (
         PARTITION BY region
         ORDER BY month
         ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
       ), 2) AS rolling_3m_avg
FROM monthly;

ROWS BETWEEN 2 PRECEDING AND CURRENT ROW gives the current row plus 2 prior rows = 3-month window. Use ROWS not RANGE here since month is a string — RANGE requires numeric/date ordering.

S6
Gaps and islands: given a table of employee login sessions (emp_id, login_date), find all continuous working streaks (no gap > 1 day).
Hard▼
WITH numbered AS (
  SELECT emp_id, login_date,
         ROW_NUMBER() OVER (PARTITION BY emp_id ORDER BY login_date) AS rn
  FROM sessions
),
grp AS (
  SELECT emp_id, login_date,
         DATE_SUB(login_date, rn) AS grp_key  -- subtract row number from date
  FROM numbered
)
SELECT emp_id,
       MIN(login_date) AS streak_start,
       MAX(login_date) AS streak_end,
       COUNT(*) AS streak_length_days
FROM grp
GROUP BY emp_id, grp_key
ORDER BY emp_id, streak_start;

The key insight: subtracting the row number from the date produces the same constant value for consecutive dates — this is the "island" identifier (grp_key).

S7
Cohort retention: given (user_id, signup_date, activity_date), compute what % of each signup-month cohort was still active in months 1, 2, and 3 after signup.
Hard▼
WITH cohorts AS (
  SELECT user_id,
         DATE_FORMAT(signup_date, 'yyyy-MM') AS cohort_month
  FROM users
),
activity AS (
  SELECT a.user_id, c.cohort_month,
         MONTHS_BETWEEN(a.activity_date, DATE(CONCAT(c.cohort_month,'-01'))) AS months_since_signup
  FROM activity_log a
  JOIN cohorts c ON a.user_id = c.user_id
  WHERE MONTHS_BETWEEN(a.activity_date, DATE(CONCAT(c.cohort_month,'-01'))) BETWEEN 0 AND 3
),
cohort_size AS (
  SELECT cohort_month, COUNT(DISTINCT user_id) AS total_users
  FROM cohorts GROUP BY cohort_month
)
SELECT a.cohort_month,
       FLOOR(a.months_since_signup) AS month_offset,
       COUNT(DISTINCT a.user_id) AS active_users,
       cs.total_users,
       ROUND(100.0 * COUNT(DISTINCT a.user_id) / cs.total_users, 1) AS retention_pct
FROM activity a
JOIN cohort_size cs ON a.cohort_month = cs.cohort_month
GROUP BY a.cohort_month, FLOOR(a.months_since_signup), cs.total_users
ORDER BY cohort_month, month_offset;

Implement SCD Type-2 logic purely in SQL. Given a current dimension table and an incoming staging table, write the MERGE statement to expire old records and insert new versions.

MERGE INTO dim_employee t

   OR (t.dept != s.dept OR t.salary != s.salary);  -- changed employees
S9
Pareto 80/20: Find the minimum set of products that together account for 80% of total revenue.
Brutal▼
WITH product_rev AS (
  SELECT product_id, SUM(revenue) AS total_rev
  FROM sales
  GROUP BY product_id
),
ranked AS (
  SELECT product_id, total_rev,
         SUM(total_rev) OVER (ORDER BY total_rev DESC
           ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS cum_rev,
         SUM(total_rev) OVER () AS grand_total
  FROM product_rev
)
SELECT product_id, total_rev,
       ROUND(100.0 * cum_rev / grand_total, 2) AS cum_pct
FROM ranked
WHERE cum_rev - total_rev < 0.8 * grand_total  -- include the product that crosses 80%
ORDER BY total_rev DESC;

The WHERE clause ensures we include the product whose addition pushes cumulative revenue over 80% — not just those already below 80%.

S10
Write a query to detect data quality issues: find all orders where (a) revenue is NULL, (b) order_date is in the future, (c) customer_id doesn't exist in the customers table. Return each violation as a separate row with a violation_type column.
Brutal▼
SELECT order_id, customer_id, order_date, revenue,
       'NULL_REVENUE' AS violation_type
FROM orders
WHERE revenue IS NULL

UNION ALL

SELECT order_id, customer_id, order_date, revenue,
       'FUTURE_ORDER_DATE' AS violation_type
FROM orders
WHERE order_date > CURRENT_DATE()

UNION ALL

SELECT o.order_id, o.customer_id, o.order_date, o.revenue,
       'INVALID_CUSTOMER_ID' AS violation_type
FROM orders o
LEFT JOIN customers c ON o.customer_id = c.customer_id
WHERE c.customer_id IS NULL

ORDER BY violation_type, order_id;

This pattern — UNION ALL of targeted checks with a violation_type label — is exactly the structure used in data quality frameworks. Each check is independently readable and extendable.
