🏗️ Cluster Architecture — Driver vs Executor
🧠 DRIVER (Brain)
✔ Runs main() method
✔ Creates SparkContext / SparkSession
✔ Plans DAG of operations
✔ Schedules tasks to executors
✔ Collects results
✗ NOT fault-tolerant (single point of failure)
✗ NOT horizontally scaled
⚙️ EXECUTOR (Worker)
✔ Runs on worker nodes
✔ Executes tasks assigned by driver
✔ Stores cached data
✔ Multiple per node possible
✔ Fault tolerant (tasks re-run)
✔ Horizontally scaled
✗ NOT synonymous with worker node

🧠 Memory Aids & Mnemonics
Visual tricks to lock in the most commonly confused concepts

Actions vs Transformations

C·S·T·F·C·H
Every action starts a data movement to the driver:
C Collect() — brings ALL data
S Show() — prints to console
T Take(n) — first n rows
F First() — row 1
C Count() — row count
H Head(n) — first n rows
🚨 Everything else = transformation (lazy!)

Wide vs Narrow — "Does it cross?"

Ask: "Does data need to cross partition boundaries?"
OperationType
filter/whereNarrow ✓
select/dropNarrow ✓
withColumnNarrow ✓
joinWide ✗
groupByWide ✗
distinctWide ✗
orderBy/sortWide ✗
repartitionWide ✗
coalesceNarrow ✓

repartition vs coalesce — "RICE"

R·e vs C·o
R Repartition = Reshuffle (full shuffle always)
+ can increase partitions
+ even distribution
C Coalesce = Combine (no full shuffle)
- can ONLY decrease
- may be uneven
💡 Reducing partitions? Use coalesce (faster)
Increasing? Must use repartition

cache() vs persist() — "C-P-S"

cache() is a shortcut for persist() with a fixed level
C cache() = MEMORY_AND_DISK (fixed, no args)
P persist(level) = choose your level
S StorageLevel.MEMORY_ONLY = memory only
🚨 cache() does NOT accept arguments!
For memory-only: use persist(StorageLevel.MEMORY_ONLY)

Broadcast Joins — "Small travels, Big stays"

Always broadcast the SMALLER DataFrame
big_df.join(broadcast(small_df), "key")
✅ Small copy sent to every executor
✅ Big DF never shuffles
❌ Never broadcast the large DF

Execution Modes — "Client Close, Cluster Far"

Where the driver lives:
C Client mode: driver on Client machine (your laptop)
C Cluster mode: driver on worker node in cluster
L Local mode: everything on one machine
🚨 "Standard mode" does NOT exist in Spark!

UDF Return Type — "int() not int"

DataTypes need parentheses — they're class instances:
❌ udf(func, IntegerType)
✅ udf(func, IntegerType())
IntegerType is a class → you need an instance → add ()

join() argument ORDER — "COT"

C · O · T
C Comrade DataFrame (second df)
O On (the key column)
T Type of join ("inner","outer"...)
df1.join(df2, "key", "inner")

1
What is the role of the Spark driver? Is it fault-tolerant?
▼

The Spark driver is the central coordinator of a Spark application. It runs the application's main() method, holds the SparkSession / SparkContext, creates the DAG (Directed Acyclic Graph) of transformations, negotiates resources with the cluster manager, and distributes tasks to executors.

🚨 The driver is NOT fault-tolerant by default — it is a single point of failure. Worker/executor failures are handled gracefully (tasks re-run), but driver failure usually kills the entire application unless external HA (e.g., Kubernetes driver HA) is configured.

The driver is also not horizontally scaled — scaling is done by adding more executors on worker nodes.

2
Explain the Spark execution hierarchy from coarsest to finest.
▼

Application → Job → Stage → Task → Slot

  • Application: Entire Spark program (holds one SparkSession)
  • Job: Triggered by an action (e.g., count(), collect())
  • Stage: Group of tasks separated by shuffle boundaries (wide transforms)
  • Task: Smallest unit of work — operates on one data partition
  • Slot: CPU thread within an executor — capacity for parallelism
A stage boundary is created by a shuffle. Narrow transformations within a stage are pipelined together without moving data across the network.
3
What is lazy evaluation in Spark and why does it matter?
▼

Lazy evaluation means that transformations (like filter, select, join) do not execute immediately when defined. Instead, Spark builds a logical plan (DAG). Execution only begins when an action is triggered (collect, count, write, etc.).

Why it matters: Spark can optimize the entire plan before running — pushing filters down, combining stages, avoiding redundant computation. This is the Catalyst optimizer at work.

The trigger is any action — not just "displaying results to the user." Even write() triggers execution.

4
Difference between repartition() and coalesce() — when to use each?
▼

repartition(n): Always performs a full shuffle. Can increase OR decrease partitions. Results in even data distribution. Use when you need more parallelism or want even partitioning.

coalesce(n): Avoids full shuffle by combining existing partitions locally. Can ONLY reduce partitions. May result in uneven distribution. Use when reducing partitions (e.g., before writing).

📏 Rule: Need to go from 8→12 partitions? Must use repartition(12). Going 8→4? Use coalesce(4) for efficiency.

coalesce(n) cannot guarantee n partitions when increasing — it effectively does nothing if target > current.

5
What is a broadcast join and when should you use it?
▼

A broadcast join sends a complete copy of the smaller DataFrame to every executor. Each executor can then perform the join locally without any data shuffling of the large DataFrame.

When to use: One DataFrame is significantly smaller than the other (e.g., a lookup table).

Always broadcast the SMALLER DataFrame:
big_df.join(broadcast(small_df), "key")
Broadcasting the large DF can cause OOM errors on executors!

spark.sql.autoBroadcastJoinThreshold controls the max size for automatic broadcasting.

6
Explain MEMORY_ONLY vs MEMORY_AND_DISK storage levels.
▼

MEMORY_ONLY: Stores all partitions in RAM. If memory is insufficient, overflow partitions are dropped and recomputed from scratch on next access. Fastest when data fits.

MEMORY_AND_DISK: Stores in RAM first. Overflow spills to disk and is read from disk on next access — no recomputation. Better when recomputation is expensive.

Choose MEMORY_AND_DISK when: reading from disk is faster than recomputing the transformation chain.

cache() = shortcut for MEMORY_AND_DISK (not MEMORY_ONLY!). For memory-only: use persist(StorageLevel.MEMORY_ONLY).

7
How do you filter a DataFrame with multiple conditions in PySpark?
▼

Use bitwise operators, not Python keywords:

# Correct: use & for AND, | for OR
df.filter( (col("sqft") <= 25000) & (col("satisfaction") >= 30) )

# WRONG: Python "and"/"or" doesn't work on columns
df.filter(col("sqft") <= 25000 and col("satisfaction") >= 30)
🔑 Always wrap each condition in parentheses due to Python's operator precedence. Use & not and, and | not or.
8
What is the difference between filter() and where() in Spark?
▼

They are exact aliases — they do precisely the same thing. Both filter rows based on a boolean condition. There is zero functional difference.

Common exam trap: picking "where is for SQL, filter is for DataFrame API" — this is false. Use whichever reads more naturally for your code.
9
How do you create a Python UDF in Spark and what is the key pitfall?
▼
from pyspark.sql.functions import udf
from pyspark.sql.types import IntegerType

# 1. Create the UDF — specify return type with ()
myUDF = udf(myFunc, IntegerType())

# 2. Apply via DataFrame API
df.withColumn("result", myUDF(col("input")))
🚨 Pitfalls: (1) Must specify return type — IntegerType() not IntegerType. (2) Return type defaults to StringType if omitted — subtle bugs! (3) Pass col("x") object, not raw string.
10
How do you register and use a SQL UDF in Spark?
▼
# Register with SQL name
spark.udf.register("MY_UDF", myFunc)

# Call using REGISTERED name, not Python func name
spark.sql("SELECT MY_UDF(col) AS result FROM table")
🚨 Common mistake: using the Python function name in SQL instead of the registered name. The SQL engine only knows the registered SQL name ("MY_UDF"), not "myFunc".
11
What causes data skew and how does Spark handle it?
▼

Data skew occurs when data is unevenly distributed across partitions — some partitions have far more data than others, causing some tasks to take much longer (stragglers).

Symptoms: Most tasks finish quickly, one or a few take forever. Overall job stalls waiting for skewed tasks.

Solutions: (1) Enable AQE with spark.sql.adaptive.skewedJoin.enabled=true — auto-splits skewed partitions. (2) Salting (add random prefix to join key). (3) Broadcast join if one side is small. (4) Increase shuffle partitions to reduce per-partition data.

Small executor memory + skewed partitions = OOM risk! More executors (more RAM per partition) reduces this risk.

12
What happens when more slots than tasks exist?
▼

When slots > tasks, some slots remain idle while the current batch of tasks runs. The job completes, but not as efficiently as possible — cluster resources are underutilized.

Spark does NOT: fail the job, auto-generate more tasks, use only one slot, or shut down executors. It simply runs with idle capacity.

Fix: Increase number of partitions (via repartition) to create more tasks and better saturate available parallelism.

13
How do you read a CSV with a custom schema in PySpark?
▼
from pyspark.sql.types import StructType, StructField, StringType, IntegerType

schema = StructType([
  StructField("name", StringType()),
  StructField("age", IntegerType())
])

df = spark.read.schema(schema).csv("path/to/file.csv")
🚨 Pass the schema OBJECT (StructType instance), NOT the string "schema". spark.read.schema("schema") will fail!
14
How do you handle null/missing values in PySpark?
▼
# Fill nulls
df.na.fill(0, "col_name") # string name, not col()

# Drop rows with ANY null
df.na.drop(how="any")

# Drop rows where ALL columns are null
df.na.drop(how="all")

# Drop for specific columns only
df.na.drop(how="any", subset=["col1", "col2"])
In na.fill(), the column name/subset argument is a string (or list of strings), NOT a col() object.
15
What is Adaptive Query Execution (AQE) in Spark 3?
▼

AQE allows Spark to re-optimize the query execution plan at runtime based on statistics collected during execution. It enables three key optimizations:

  • Coalescing small partitions after shuffle (spark.sql.adaptive.coalescePartitions.enabled)
  • Skew join handling — splits skewed partitions (spark.sql.adaptive.skewedJoin.enabled)
  • Converting sort-merge joins to broadcast joins when runtime data is smaller than threshold
AQE is a game-changer because it fixes problems that static planning cannot foresee (actual data size, skew).
16
What is the difference between union() and unionByName()?
▼

union(): Combines DataFrames by column POSITION — the first column of df1 is appended with the first column of df2, regardless of column names. Schemas must have same number of columns.

unionByName(): Combines DataFrames by column NAME — aligns matching column names. Handles different column ordering. Can allow missing columns with allowMissingColumns=True.

Use union() when schemas are identical. Use unionByName() when column names should be the alignment key, or when DataFrames might have different column ordering.
17
How does Spark achieve fault tolerance?
▼

Spark achieves fault tolerance through RDD lineage (the DAG of operations). If a partition is lost (e.g., executor failure), Spark can recompute it from the original data by replaying the transformation chain.

  • Failed tasks are automatically re-run on another executor
  • Cached data on failed nodes is recomputed
  • Data that doesn't fit in memory spills to disk
  • Supports loss of any set of worker nodes
🚨 Driver failure is NOT automatically recovered! Only worker/executor failures are fault-tolerant by default.
18
What is a partition in Spark and what is the ideal partition size?
▼

A partition is a collection of rows that fit on a single machine in the cluster. It's the fundamental unit of parallelism — each task processes exactly one partition.

General guidelines: Target 128MB-256MB per partition for processing. For writing, smaller is OK but avoid too many tiny files. Too few partitions = under-parallelism. Too many = scheduling overhead.

The number of partitions determines the degree of parallelism. More partitions → more tasks → more parallel execution (up to available slots).

19
What is the correct way to perform a cross join in PySpark?
▼
# Correct: use dedicated crossJoin() method
result = df1.crossJoin(df2)

# WRONG: join() with "cross" string fails
result = df1.join(df2, "cross") # ❌ Error
Cross join has no join key — it produces the Cartesian product (every row × every row). Use crossJoin() — a dedicated method. The join() method's "cross" type argument also works but crossJoin() is explicit.
20
Explain spark.sql.shuffle.partitions vs spark.default.parallelism.
▼

spark.sql.shuffle.partitions: Controls the number of partitions created AFTER a shuffle operation in Spark SQL (DataFrame/Dataset API) — used by join, groupBy, agg, etc. Default: 200.

spark.default.parallelism: Controls default partitions for RDD operations (map, flatMap, etc.) — NOT for DataFrame/SQL shuffle operations.

For optimizing join/groupBy performance → change spark.sql.shuffle.partitions. Setting spark.default.parallelism does NOT fix slow joins!

Exercise 2 Aggregations & GroupBy 
from pyspark.sql.functions import count, mean, sum, max, min, approx_count_distinct, countDistinct

# ✅ GroupBy + count — returns one row per group
df.groupBy("division").count()

# ✅ GroupBy multiple columns
df.groupBy("division", "category").count()

# ✅ agg() for multiple aggregations — mean() needs col()
df.agg(
    mean(col("sqft")).alias("avgSqft"),
    max(col("sqft")).alias("maxSqft"),
    count("*").alias("total")
)

# ✅ Approximate distinct count — higher rsd = faster, less accurate
df.agg(approx_count_distinct(col("division"), 0.15).alias("approxDistinct"))
#   ^-- 0.15 rsd = fastest; 0.0 = most accurate (but slow)

# ✅ Exact distinct count
df.agg(countDistinct(col("division")).alias("exactDistinct"))

# ✅ Summary stats — pass string, NOT col()
df.describe("sqft")   # ✅ Correct
# df.describe(col("sqft"))  # ❌ WRONG — col() not accepted

# ✅ Sort ascending / descending
df.orderBy("division")          # ascending (default)
df.orderBy(col("division").desc())  # descending
df.orderBy("division", ascending=False)  # descending

Exercise 4 String Functions & Date Operations 
from pyspark.sql.functions import split, explode, regexp_replace, from_unixtime, month, dayofyear

data = [(1, "Electronics,Books,Toys", "HIGH_LARGE", 1228348800)]
df = spark.createDataFrame(data, ["id", "categories", "storeType", "openDate"])

# ✅ Split string into array (split is standalone function)
df = df.withColumn("categoryArr", split(col("categories"), ","))
# Result: ["Electronics","Books","Toys"]

# ✅ Get element from array — zero-indexed
df = df.withColumn("firstCat", split(col("categories"), ",")[0])

# ✅ Explode array into rows — multiplies row count!
df.withColumn("category", explode(col("categoryArr")))

# ✅ Replace regex pattern (3 required args)
df.withColumn("storeType", regexp_replace(col("storeType"), "_", "-"))

# ✅ Split storeType into two columns
(df.withColumn("valueType", split(col("storeType"), "_")[0])
   .withColumn("sizeType",  split(col("storeType"), "_")[1]))

# ✅ Unix timestamp → formatted string (2 args only)
df.withColumn("dateStr", from_unixtime(col("openDate"), "EEEE, MMM d, yyyy h:mm a"))

# ✅ Unix int → Timestamp → month (must cast first)
(df.withColumn("ts", col("openDate").cast("Timestamp"))
   .withColumn("month", month(col("ts"))))

# ❌ WRONG — dayofyear() cannot take integer column directly
# df.withColumn("doy", dayofyear(col("openDate")))  # ERROR

Exercise 5 UDFs, Caching & Write Operations 
from pyspark.sql.functions import udf
from pyspark.sql.types import IntegerType, StringType
from pyspark.storagelevel import StorageLevel

# ===== UDFs =====

def assessPerformance(score):
    return 1 if score >= 30 else 0

# ✅ Create UDF — specify return type with ()
assessUDF = udf(assessPerformance, IntegerType())

# ✅ Apply via DataFrame API
df.withColumn("result", assessUDF(col("satisfaction")))

# ✅ Register for SQL use
spark.udf.register("ASSESS_PERF", assessPerformance)

# ✅ SQL must use REGISTERED name, not Python function name
spark.sql("SELECT ASSESS_PERF(satisfaction) AS result FROM stores")

# ===== CACHING =====

# ✅ Cache at MEMORY_AND_DISK (default)
df.cache().count()

# ✅ Memory-only — use persist with StorageLevel
df.persist(StorageLevel.MEMORY_ONLY).count()

# ❌ WRONG — cache() doesn't take arguments
# df.cache(StorageLevel.MEMORY_ONLY)  # Error

# ===== WRITING DATA =====

# ✅ Write as Parquet — overwrite
df.write.mode("overwrite").parquet("/path/output")

# ✅ Write partitioned by column
df.write.partitionBy("division").parquet("/path/output")

# ✅ Write as CSV
df.write.csv("/path/output")

# ✅ Write as JSON
df.write.json("/path/output")

# ✅ Repartition config for faster joins
spark.conf.set("spark.sql.shuffle.partitions", "32")

# ===== CREATING DataFrames =====
years = [2020, 2021, 2022]
df_years = spark.createDataFrame(years, IntegerType())  # Pass instance, not class
