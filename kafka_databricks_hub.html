Core Components essentials

Kafka is a distributed, durable commit log. Records are appended to ordered, immutable partitions. Partitions live on brokers; multiple brokers form a cluster. Each partition has one leader that handles all reads/writes; followers replicate asynchronously.

ComponentRoleKey Property
TopicLogical stream name, split into N partitionsRetained after consumption
PartitionOrdered, append-only log; unit of parallelismRecords never deleted by consumers
OffsetMonotonically increasing ID within a partitionUnique position per partition
BrokerSingle Kafka server hosting partition replicasOne leader + N-1 follower replicas
ISRIn-Sync Replicas — caught up to leaderOnly ISR members can become leaders
ControllerElected broker managing cluster membershipOne per cluster at all times
KRaft vs ZooKeeper kafka 3.3+
AspectZooKeeper ModeKRaft Mode
MetadataExternal ZooKeeper cluster (3–5 nodes)Built-in Raft log on Kafka controller nodes
Partition scale~200K partitionsMillions of partitions
Ops complexityTwo systems to manageSingle unified system
Failover timeSeconds (ZK re-election)Milliseconds (Raft already has full log)
StatusRemoved in Kafka 4.0GA since 3.3; default since 3.7
✓ All new deployments should use KRaft. ZooKeeper support is completely removed in Kafka 4.0.
Log Compaction vs Log Deletion
cleanup.policy=delete

Purges old log segments by time (retention.ms) or size. Every event retained until expiry. Best for event streams, metrics, time-series logs.

cleanup.policy=compact

Retains only the latest value per key. Old values replaced with tombstones (null). Best for CDC tables, config stores, Kafka Streams KTables.

Batching Internals

Records enter the RecordAccumulator (per-partition memory queue). The I/O thread flushes when batch reaches batch.size bytes or after linger.ms.

linger.ms
0 ms
default → set 5–20 ms for better batch fill
compression.type
lz4

Consumers — Deep Dive
Consumer Groups & Partition Assignment

Each partition is consumed by exactly one consumer in a group. Multiple groups can independently read the same topic. The group coordinator broker manages membership; the group leader runs the assignment algorithm.

StrategyBehaviorBest For
RangeAssignorContiguous ranges per topicSimple pipelines
RoundRobinAssignorEven distribution alphabeticallyBetter balance
StickyAssignorMinimises movement on rebalanceStateful consumers
CooperativeStickyAssignorIncremental — only moves what must moveProduction — no stop-the-world
✓ Use CooperativeStickyAssignor for Kafka Streams and Spark — prevents full stop-the-world rebalance spikes.
Offset Management

Offsets stored in __consumer_offsets. Committing offset N means "I've processed up to N−1; next fetch starts at N".

// Controlled at-least-once: commit after successful processing
while (true) {
  var records = consumer.poll(Duration.ofMillis(100));
  for (var r : records) { process(r); }
  consumer.commitSync();  // blocks until broker confirms
}
⚠ auto.commit=true can cause silent data loss if consumer crashes between commit and processing completion. Always use manual commit for DE pipelines.

Kafka Streams
KStream vs KTable vs GlobalKTable
KStream

Unbounded event stream. Each record independent. Think INSERT-only log. Stateless or stateful ops.

KTable

Partitioned changelog. Each key has latest value. Think CDC-driven lookup table. Backed by RocksDB.

GlobalKTable

Full topic replicated to every task. Foreign-key joins without repartitioning. Best for small reference data.

// Full DSL example: filter → enrich via join → window aggregate
StreamsBuilder b = new StreamsBuilder();
KStream<String,Order> orders = b.stream("orders");

// Stateless
KStream<String,Order> high = orders
  .filter((k,v) -> v.getAmount() > 500)
  .mapValues(v -> enrichWithTax(v));

// Stateful: 5-min tumbling window count
KTable<Windowed<String>,Long> counts = orders
  .groupByKey()
  .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5)))
  .count(Materialized.as("order-counts"));

// KStream-KTable join: enrich event with user profile
KTable<String,User> users = b.table("users");
KStream<String,Enriched> enriched =
  orders.join(users, (o,u) -> new Enriched(o,u));

Stateless vs Stateful Processing
Stateless Processing simple

Each event is processed independently. No memory of past events. Output depends only on the current input record.

Examples
  • Filter events where amount > 1000
  • Map/transform field names
  • Route to topic by event type
  • Enrich from static lookup table
# Stateless: filter + transform
stream
  .filter("amount > 1000")
  .selectExpr(
    "user_id",
    "amount * 1.18 as amount_with_tax"
  )
✓ Horizontally scalable. Any consumer instance can handle any message. Restarts are safe — no state recovery needed.
Stateful Processing complex

Processing depends on accumulated state from past events. Requires a persistent state store (RocksDB, checkpoints, Delta).

Examples
  • Running totals per user
  • Windowed aggregations (sum, count)
  • Stream-stream joins
  • Deduplication
  • User session analytics
# Stateful: 5-min count per user (Spark SS)
stream
  .groupBy("user_id")
  .agg(count("*").alias("event_count"))
  .writeStream
  .outputMode("update")
  .start()
⚠ Key challenge: state must be recovered after restarts. All events for a key must route to the same partition. State can grow unbounded without TTL/eviction.

Streaming vs Batch
Streaming unbounded
  • Processes events continuously as they arrive
  • Latency: milliseconds to seconds
  • Unbounded, infinite data source
  • Results updated incrementally
  • Requires watermarks for event-time correctness
  • Higher operational complexity
USE CASES
Fraud detection Live dashboards IoT alerting CDC pipelines
Batch bounded
  • Processes bounded datasets on a schedule
  • Latency: minutes to hours
  • Bounded, finite data input
  • Results computed from scratch each run
  • Simpler mental model and debugging
  • Efficient for large-scale transforms
USE CASES
Daily reports ETL to warehouse ML training Historical backfill
Micro-batch — Spark's Middle Ground default spark SS mode

Spark Structured Streaming uses micro-batch by default: streaming is approximated by running many tiny batches (e.g., every 30 seconds). Combines streaming API semantics with batch engine efficiency.

Decode Kafka Payload & Write to Delta Lake
from pyspark.sql.functions import col, from_json, from_avro
from pyspark.sql.types import StructType, StringType, LongType

# Define schema for JSON payload
schema = (StructType()
  .add("user_id", LongType())
  .add("event", StringType())
  .add("ts", LongType()))

# Parse binary Kafka value
parsed = (df
  .selectExpr("CAST(key AS STRING)", "CAST(value AS STRING)", "timestamp")
  .select(from_json(col("value"), schema).alias("d"), "timestamp")
  .select("d.*", "timestamp"))

# Write to Delta Lake bronze table
query = (parsed.writeStream
  .format("delta")
  .outputMode("append")
  .option("checkpointLocation", "/mnt/checkpoints/user-events")
  .trigger(processingTime="30 seconds")   # micro-batch
  .table("bronze.user_events"))
✓ Delta Lake checkpointing provides exactly-once semantics. The checkpoint stores committed Kafka offsets and Delta transaction IDs atomically.

foreachBatch — Custom Sink Logic
def upsert_to_silver(batch_df, batch_id):
    """Idempotent upsert from Kafka CDC stream into silver Delta table."""
    from delta.tables import DeltaTable
    dt = DeltaTable.forName(spark, "silver.orders")
    (dt.alias("t")
      .merge(batch_df.alias("s"), "t.order_id = s.order_id")
      .whenMatchedUpdateAll()
      .whenNotMatchedInsertAll()
      .execute())

query = (parsed.writeStream
  .foreachBatch(upsert_to_silver)
  .option("checkpointLocation", "/chk/silver-orders")
  .start())

DLT Reading Kafka — Bronze Layer
-- DLT SQL: ingest Kafka topic as streaming bronze table
CREATE OR REFRESH STREAMING TABLE bronze_user_events
COMMENT "Raw user events from Kafka topic user-events"
AS SELECT
  CAST(key AS STRING) AS event_key,
  from_json(CAST(value AS STRING),
    'user_id LONG, event STRING, amount DOUBLE, ts LONG') AS payload,
  timestamp AS kafka_timestamp,
  partition AS kafka_partition,
  offset AS kafka_offset
FROM stream(read_kafka(
  bootstrapServers => 'broker1:9092',
  subscribe => 'user-events'
));
DLT Silver Layer — Data Quality with Expectations
-- DLT Python: silver layer with expectations (data quality constraints)
import dlt
from pyspark.sql.functions import col, expr

# Expectation: DROP rows violating constraint
@dlt.table(name="silver_user_events", comment="Validated events")
@dlt.expect_or_drop("valid_amount", "payload.amount > 0")
@dlt.expect_or_drop("valid_user", "payload.user_id IS NOT NULL")
@dlt.expect("no_future_events", "payload.ts <= unix_timestamp()")
def silver_user_events():
    return (dlt.read_stream("bronze_user_events")
      .select(
        col("payload.user_id"),
        col("payload.event"),
        col("payload.amount"),
        expr("TIMESTAMP(payload.ts / 1000)").alias("event_time"),
        col("kafka_timestamp")
      ))
DLT Gold Layer — Aggregated Business Metrics
-- DLT SQL: Gold — hourly revenue materialized view
CREATE OR REFRESH MATERIALIZED VIEW gold_hourly_revenue
COMMENT "Hourly revenue aggregation from validated events"
AS
SELECT
  date_trunc('hour', event_time) AS hour,
  user_id,
  event,
  SUM(amount) AS total_revenue,
  COUNT(*) AS event_count
FROM LIVE.silver_user_events
GROUP BY 1, 2, 3;

Avro + Schema Registry in Databricks
# from_avro() with Confluent Schema Registry
from pyspark.sql.functions import col
from pyspark.sql.avro.functions import from_avro

schema_registry_options = {
  "mode": "PERMISSIVE",
  "schema.registry.address": "https://schema-registry:8081"
}

# Strip the 5-byte magic header before deserializing
avro_df = (df
  .select(
    from_avro(
      col("value"),    # binary Kafka value
      "user-events-value",   # subject name in Schema Registry
      schema_registry_options
    ).alias("data")
  )
  .select("data.*"))
Best Practices Checklist
DATABRICKS SIDE
  • Always set checkpointLocation — stores offset state
  • Use maxOffsetsPerTrigger to throttle bursts
  • Use startingOffsets=latest in production
  • Monitor with Spark Streaming UI → lag trends
  • Prefer DLT over raw Spark SS for new pipelines
  • Use Unity Catalog for Delta table governance
KAFKA SIDE
  • Partition count ≥ Databricks executor count
  • Use SASL_SSL for secure connections
  • Enable Schema Registry for schema evolution
  • Set min.insync.replicas=2 with acks=all
  • Monitor consumer group lag for Databricks group
  • Use log compaction for CDC topics feeding DLT
